## 修改前行为
1. **MDAS `_fetch_intraday_with_backfill`**：`连续2轮无新增` 直接判定为 `history_exhausted`，对长期停牌股票不安全（停牌>180 天但更早有 4000 根历史时会错误终止）
2. **因子版本首次运行**：`find_stale_version_instruments` 定义但未接入生产调用链；生产 DB 中 8272 只 active 股票的 `factor_algorithm_version` 字段均为 NULL；无安全 bootstrap 机制；若未来直接接入 `find_stale_version_instruments` 会一次性把 8272 只股票全部识别为 stale

## 修改后行为

### CP-V3-A3: MDAS 长期停牌边界修正
1. **listing_date 作为历史下界**：只有查询到达 instrument 上市日（或 `_MIN_A_SHARE_DATE=1990-01-01`）仍不足 required_count 时，才允许 `history_exhausted=True`
2. **no_progress 只扩大步长**：no_progress 不再终止回补，而是有界递增扩展步长（90→180→360→720 天，上限 720）
3. **max_rounds 未到边界→INPUT_CONTRACT_VIOLATION**：达到 `_MAX_BACKFILL_ROUNDS=10` 但未到达 listing_date 且不足 required_count 时，返回 `history_exhausted=False`，调用方判定为 `unavailable/INPUT_CONTRACT_VIOLATION`
4. **调用方传入 listing_date**：`_fetch_intraday_with_backfill` 新增 `listing_date` 参数；调用方通过 `_get_listing_date` 查询 instruments 表获取上市日

### CP-V3-D2: 因子版本首次运行安全 bootstrap
1. **`bootstrap_factor_version_baseline` 函数**：基于已有 dry_run 审计证据，对 consistent 且 NULL 版本的股票写入版本基线（不重算因子）
2. **跳过 needs_rebuild/degraded**：needs_rebuild 由 after-close 重建后自动 stamp；degraded 需先回补数据
3. **安全保证**：不触发全市场重建、不修改因子数据、幂等（只处理 NULL）、可中断（分批 commit）、可恢复、有限流
4. **`find_stale_version_instruments` docstring 警告**：明确标注"未 bootstrap 前禁止接入 after-close 流程"

## 8272 NULL 首次运行回答

### 1. 部署后首次盘后是否会识别 8272 只全部 stale？
**否**。`find_stale_version_instruments` 未接入 after-close 流程。after-close 使用的是 `_audit_and_rebuild_factors`（基于 dry_run hash 比对，不基于版本字段）。

### 2. 是否会自动触发全市场重建？
**否**。`_audit_and_rebuild_factors` 的 `dry_run` 审计全市场（基于 hash 比对），只重建审计发现不一致的股票（`rebuild_batch(batch_size=10)`）。8272 NULL 版本字段不会触发全市场重建。

### 3. 每批大小、失败恢复和资源峰值
- **审计阶段**：`dry_run(batch_size=50, max_mismatches=20)` — 全市场只读审计，分批 50 只
- **重建阶段**：`rebuild_batch(batch_size=10)` — 串行重建不一致股票，每只独立事务
- **失败恢复**：每只股票独立事务，失败回滚不影响其他股票；失败清单写入 `job_run_event` 留下诊断痕迹
- **资源峰值**：审计阶段只读（无写入峰值）；重建阶段串行（无并发峰值）；2026-07-19 实测 614 只重建耗时 3074.9s ≈ 51 分钟

### 4. "闭环完成"声明修正
**因子版本闭环未完成**。`find_stale_version_instruments` 未接入生产调用链，所以"基于版本识别影响集"未启用。当前 after-close 流程基于 dry_run hash 比对（非版本驱动）。`stamp_factor_reconciliation_version` 已接入 `_rebuild_single` 成功路径，所以重建成功的股票会写入版本字段，但这些字段目前无消费者。

## 安全 bootstrap 设计

### 核心原则
- 已有审计证据确认一致的股票：只写版本基线，不重复重算
- 无充分证据的股票：按小批次审计
- 只有确认不一致的股票：进入重建（由 `_audit_and_rebuild_factors` 处理）
- 每批可恢复、可停止、有限流
- 不得在本轮修改生产 DB

### 实现：`bootstrap_factor_version_baseline`
1. 获取 `dry_run_plan`（传入或内部调用 `dry_run` 全市场审计）
2. 查询所有 `factor_algorithm_version IS NULL` 的 active 股票
3. 排除 `dry_run_plan.items`（needs_rebuild）和 `dry_run_plan.degraded_symbols`
4. 对剩余股票（consistent 且 NULL）写入版本基线
5. 分批 commit（每批 `batch_size` 只），可中断、可恢复

### 安全保证
- **不触发全市场重建**：只写版本字段，不调用 `rebuild_factor_series`
- **不修改因子数据**：只更新 `factor_algorithm_version` 等版本字段
- **幂等**：`WHERE factor_algorithm_version IS NULL`，已写入的跳过
- **可中断**：分批 commit，中断后可重跑（已写入的跳过）
- **可恢复**：失败批次不影响已 commit 的批次（`_stamp_batch` 失败时 rollback）
- **有限流**：`batch_size` 控制每批数量，串行执行

### 使用场景
部署后首次盘后前，手动执行一次 bootstrap（不在本轮执行，不修改生产 DB）：
```python
async with AsyncSessionLocal() as session:
    result = await bootstrap_factor_version_baseline(session, batch_size=100)
    # result["stamped"] = 写入版本基线的股票数
    # result["skipped_needs_rebuild"] = 跳过的 needs_rebuild 数量
    # result["skipped_degraded"] = 跳过的 degraded 数量
```
