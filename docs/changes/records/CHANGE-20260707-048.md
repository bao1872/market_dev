# CHANGE-20260707-048: Snapshot Run Gate + Instrument-first Backfill

## 变更编号
CHANGE-20260707-048

## 任务名称
Feature Snapshot 运行态安全 + instrument-first backfill 重构

## 需求出处
PR #38 合并部署后用户要求进入下一 PR：
- 修复 scripts 目录未进 Docker 镜像的问题；
- 清理 PR #38 smoke test 写入的 25 行非完整快照；
- 新增 snapshot date/run 级别成功标记，watchlist 只读成功完成的 snapshot 日期；
- 将 backfill 从 date-first 优化为 instrument-first；
- 仍保持小步验证，不全量生产回补。

## 修改前行为
1. `backend/scripts/` 目录未 COPY 到 Docker 镜像，容器内无法直接运行 `python -m scripts.feature_snapshot_backfill`；
2. PR #38 smoke test 在生产 `stock_feature_snapshots` 表写入了 25 行 2026-07-07 的非完整快照（无 succeeded run 标记）；
3. watchlist 只通过 `expected_snapshot_trade_date` 查询 snapshot 是否存在，无法区分"计算中"（running）/"已失败"（failed）/"已成功"（succeeded），可能读取半成品行；
4. backfill 脚本为 date-first 架构：每个交易日遍历全市场，每只股票重复 fetch 1d/15m bars，全量回补非常重。

## 修改后行为
1. `backend/Dockerfile` 新增 `COPY scripts ./scripts`，容器内可直接运行 `python -m scripts.feature_snapshot_backfill --help`；
2. 新增 `stock_feature_snapshot_runs` 表（partial unique index 仅约束 `status='running'`，允许 failed run 与新 retry 并存）；
3. `after_close` / `backfill` 开始时创建 `running` run（独立 session + commit），成功时 `finish_snapshot_run(status='succeeded')` 写 `published_at`，失败时 `finish_snapshot_run(status='failed')` 不写 `published_at`；
4. `/watchlist/monitor-status` 通过 `_has_succeeded_snapshot_run` 判断 `expected_snapshot_trade_date` 是否有 `succeeded` run，无 succeeded run 时返回 `WAITING_SNAPSHOT` 或 `NO_SNAPSHOT`；
5. backfill 脚本重构为 instrument-first：每只股票每周期只调用一次 `load_instrument_bars`，内存中按 `trade_date` slice，复用 `compute_feature_snapshot_for_date(primary_bars=..., secondary_bars=...)` 已有入参；
6. backfill 新增 `--symbols` / `--limit-instruments` 参数支持小样本验证；
7. backfill run gate：每个 trade_date 创建 `succeeded`/`failed` run，`--resume` 跳过已存在 snapshot 且所属日期有 `succeeded` run 的行（双重过滤）。
