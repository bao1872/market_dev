# 安全规范

## 1. 密钥管理

### 1.1 密钥清单

事实源：`backend/.env.example` + `docker-compose.prod.yml`

| 密钥 | 用途 | 存储位置 | 轮换频率 |
|------|------|---------|---------|
| `JWT_SECRET` | JWT 签名 | env 文件 | 季度 |
| `SECRET_MASTER_KEY` | Secret 加密主密钥（Fernet 派生） | env 文件 | 季度 |
| `DATABASE_URL` 中的密码 | PostgreSQL 业务用户密码 | env 文件 | 半年 |
| 飞书 `app_secret` | 飞书平台应用凭证 | DB（加密存储） | 应用变更时 |
| 飞书 `sign_secret` | Webhook 签名密钥 | DB（加密存储） | 应用变更时 |

### 1.2 密钥存储

- **生产 env 文件**：`/etc/market-dev/market.env`，权限 `600`，仅 root 可读
- **生产 Python 配置文件**：通过 `CONFIG_FILE` 环境变量指向 `/etc/market-dev/config.production.py`（权限 `600`，仅 root 可读），由 `config.py::_load_py_config` 加载
- **禁止**：将真实密码写入代码、提交到 Git、记录到日志
- **禁止**：在 `.env.example` / `config.example.py` 中填写真实密码
- **禁止**：在 `config.py` 中保留 `es123456` / `postgres` 等默认密码 fallback
- **禁止**：通过 `ALTER USER` 修改数据库密码（密码由 DBA 在初始化时设置，仅通过 env/CONFIG_FILE 注入）
- **DB 中的密钥**：通过 `encrypt_secret` 加密存储，读取时通过 `decrypt_secret` 解密

**config.py 职责分离**（事实源：`backend/app/config.py`）：
- `config.py`：仅定义 `Settings` 结构、加载规则、启动硬校验。**入库**。
- `config.local.py` / `config.test.py`：开发/测试环境实际值。**不入库**（`.gitignore` + `.dockerignore`）。
- `config.production.py`：生产环境实际值，通过 `CONFIG_FILE` 挂载。**不入库**。
- `config.example.py`：必需字段示例（占位符）。**入库**。

### 1.3 密钥派生

事实源：`backend/app/core/security.py`

**JWT 签名**：
```python
jwt.encode(payload, settings.jwt_secret, algorithm=settings.jwt_algorithm)
# 默认算法：HS256
```

**Fernet 加密**（用于 Secret 字段）：
```python
# 从 SECRET_MASTER_KEY 派生 Fernet 密钥
digest = hashlib.sha256(settings.secret_master_key.encode()).digest()
fernet_key = base64.urlsafe_b64encode(digest)
# Fernet 要求 32 字节密钥 urlsafe base64 编码
```

### 1.4 密钥轮换流程

**JWT_SECRET 轮换**：
1. 在 env 文件中新增 `JWT_SECRET_NEW=<new-secret>`
2. 修改 `app/core/deps.py` 支持 dual secret 验证（新旧都接受）
3. 重启 backend
4. 等待所有 access token 过期（最长 1 小时）
5. 移除旧 `JWT_SECRET`，将 `JWT_SECRET_NEW` 重命名为 `JWT_SECRET`
6. 重启 backend

**SECRET_MASTER_KEY 轮换**：
1. 编写迁移脚本：遍历所有加密字段，用旧密钥解密，用新密钥加密
2. 在 env 文件中设置新密钥
3. 执行迁移脚本
4. 重启 backend

> **重要**：SECRET_MASTER_KEY 轮换需要先迁移历史数据，否则无法解密。

## 2. 数据库角色

### 2.1 角色定义

| 角色 | 权限范围 | 使用场景 |
|------|---------|---------|
| `postgres` | 超级管理员，全部权限 | DBA 维护操作 |
| `bz` | `bz_stock` / `bz_stock_test` 全部权限 | 应用业务读写 |

### 2.2 角色创建

```sql
-- 使用 postgres 管理员执行（密码通过环境变量或交互式输入，禁止写入文档）

CREATE DATABASE bz_stock OWNER bz;
CREATE DATABASE bz_stock_test OWNER bz;
GRANT ALL PRIVILEGES ON DATABASE bz_stock TO bz;
GRANT ALL PRIVILEGES ON DATABASE bz_stock_test TO bz;
```

> **禁止**：在文档、脚本、代码中硬编码真实数据库密码。密码仅通过 `/etc/market-dev/market.env`（权限 600）或 DBA 交互式输入提供。

### 2.3 连接限制

- **生产库**：仅 127.0.0.1 监听，不暴露到公网
- **测试库**：容器内 5432 → 宿主机 5433，仅本地访问
- **应用连接**：通过 `DATABASE_URL` 环境变量，使用 `bz` 角色

### 2.4 最小权限原则

- 应用仅使用 `bz` 角色，不使用 `postgres` 超级管理员
- 测试库与生产库物理隔离（不同数据库 + 不同端口）
- 禁止应用账号执行 DDL（迁移由 Alembic 单独执行）

## 4. 认证与授权

### 4.1 JWT Token 类型

事实源：`backend/app/core/deps.py` + `security.py`

| Token 类型 | TTL | 用途 | 限制 |
|-----------|-----|------|------|
| `access` | 1 小时 | API 认证 | 用户状态必须 active |
| `refresh` | 7 天 | 刷新 access token | 仅用于 `/auth/refresh` |
| `capture` | 短期 | 截图模式 | 仅 `/stock/{symbol}?capture=feishu` |

### 4.2 RBAC 实现

事实源：`backend/app/core/deps.py`

```python
# 角色检查依赖工厂
require_roles("admin")  # 任一匹配即通过

# 用法
@router.post("/admin/invite-codes", dependencies=[Depends(require_roles("admin"))])
```

### 4.3 安全约束

- **用户 ID 注入**：私有资源的 `user_id` 由认证上下文注入，不接受客户端传入
- **状态检查**：`status != active` 的用户拒绝访问（403）
- **常量时间比较**：密码验证使用 `bcrypt.checkpw`（避免时序攻击）
- **密码长度限制**：bcrypt 限制 72 字节，超过截断

### 4.4 密码哈希

事实源：`backend/app/core/security.py`

```python
# 哈希
hashed = bcrypt.hashpw(plain.encode("utf-8")[:72], bcrypt.gensalt())

# 验证（常量时间比较）
bcrypt.checkpw(plain.encode("utf-8")[:72], hashed.encode("utf-8"))
```

**约束**：
- 直接使用 `bcrypt` 库（避免 passlib 1.7.4 与 bcrypt 5.x 兼容性问题）
- 自动处理盐与版本前缀（`$2b$`）
- 密码超过 72 字节时截断

## 10. 安全事件响应

### 10.1 密钥泄露

1. 立即轮换泄露的密钥（参考第 1.4 节）
2. 检查日志中是否有异常访问
3. 通知所有用户重新登录（轮换 JWT_SECRET 后强制下线）
4. 复盘泄露原因，修复流程

### 10.2 数据库被入侵

1. 立即修改 `bz` 用户密码
2. 检查 `scheduler_job_runs` / `worker_heartbeats` 是否有异常记录
3. 检查 `users` 表是否有未授权用户
4. 检查 `notification_channels` 是否有可疑渠道
5. 从备份恢复（如必要）

### 10.3 用户账号被入侵

1. 禁用账号：`UPDATE users SET status = 'disabled' WHERE email = 'xxx';`
2. 撤销该用户的所有 token（轮换 JWT_SECRET，影响所有用户）
3. 通知用户重置密码
4. 检查该用户的操作记录

## 11. 权限架构 V2 安全修复（Phase 3）

本节记录权限架构 V2（Phase 2）落地过程中发现并修复的 5 项安全问题。根因是早期实现为图便捷绕过了统一的 JWT 认证与 RBAC 依赖，Phase 3 将其全部收敛回 `get_current_active_user` / `require_roles("admin")` 单一认证链路，与第 4 节"认证与授权"约束对齐。

### 11.1 X-User-Id 认证移除

事实源：`backend/app/api/notifications.py`

- **问题**：notifications 端点通过私有函数 `_get_user_id` 从请求头 `X-User-Id` 读取用户身份，无任何签名校验。
- **风险等级**：严重 — 任意客户端可伪造 `X-User-Id: <他人 user_id>` 冒充任意用户读取消息、操作渠道。
- **修复**：删除 `_get_user_id` 函数，所有端点改用 `Depends(get_current_active_user)` 依赖注入，用户身份由 JWT Bearer token 解析得到（与第 4.3 节"用户 ID 注入"约束一致：私有资源的 `user_id` 由认证上下文注入，不接受客户端传入）。
- **验证**：75 个 notifications 相关测试通过。

### 11.2 通知渠道所有权校验

事实源：`backend/app/services/notification_service.py` + `backend/app/api/notifications.py`

- **问题**：`verify_channel` 与 `test_channel` 仅按 `channel_id` 查询渠道，不校验调用者是否为渠道所有者。
- **风险等级**：高 — 用户可调用他人渠道的验证/测试接口，触发他人飞书 Bot 投递或泄露渠道配置状态。
- **修复**：两个函数新增 `user_id` 参数，校验 `channel.user_id == user_id`，不匹配抛 `ChannelOwnershipError`（HTTP 403）。配合 11.1 的 `get_current_active_user`，`user_id` 由认证上下文注入，不接受客户端传入。
- **验证**：79 个渠道所有权相关测试通过。

### 11.3 capture token 隔离

事实源：`backend/app/core/deps.py`

- **问题**：`get_current_user` 同时接受 `access` 与 `capture` 两种 token 类型，导致截图用的短期 capture token 可通过 `Authorization: Bearer <capture_token>` 调用任意 API。
- **风险等级**：中 — capture token 通过 URL query parameter 传递（`/stock/{symbol}?capture=feishu`），易被日志/Referer 泄露，若可越权将放大泄露面。
- **修复**：`get_current_user` 收紧为仅接受 `access` token（`payload.type != "access"` 即拒 401）。capture token 仅通过 URL query parameter 用于截图端点，不经过 `get_current_user` 依赖链路。
- **与第 4.1 节的一致性**：第 4.1 节已明确 capture token "仅 `/stock/{symbol}?capture=feishu`"，本修复在代码层强制落实该约束。
- **验证**：3 个 capture token 隔离测试通过（含 capture token 经 Authorization header 访问 `/me/access` 返回 401 的用例）。

### 11.4 admin 路由权限保护

事实源：`backend/app/api/strategies.py`

- **问题**：3 个 admin 端点完全无认证，任意未认证用户可创建/发布/归档策略：
  - `POST /admin/strategies`（创建策略 + 草稿版本）
  - `POST /admin/strategies/{key}/versions/{version}/release`（发布版本，draft → released）
  - `POST /admin/strategies/{key}/versions/{version}/archive`（归档版本，released → archived）
- **风险等级**：严重 — 策略版本是策略资产的权威来源，未认证发布/归档会污染线上策略计算口径。
- **修复**：三个端点均添加 `_user=Depends(require_roles("admin"))`（与第 4.2 节 RBAC 实现一致）。
- **验证**：6 个 admin 端点权限测试通过（未认证 / 非 admin 角色 / admin 角色三类用例）。

### 11.5 前端 401/403 分离

事实源：`frontend/src/api/client.ts`

- **问题**：axios 响应拦截器未区分 401 与 403，403（权限不足）错误可能误走 401 的 token 刷新流程，导致清除 token / 跳转登录页。
- **风险等级**：低 — 不会造成数据损坏，但用户体验异常（权限不足却被踢下线）。
- **修复**：响应拦截器新增 403 显式处理分支：toast 提示 + reject，不清除 token、不跳转登录页，与 401 refresh 流程完全隔离。
- **与后端契约**：后端 403 语义为"用户状态非 active / 权限不足"（见 `docs/API与事件契约.md` 第 7 节），前端 403 分支与该语义对齐；401 仍走 token 刷新。
- **验证**：36 个前端契约测试通过。
