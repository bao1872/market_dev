## 3. 测试规范

### 3.1 测试库要求

事实源：`backend/tests/conftest.py`

- **强制**：测试必须在 `APP_ENV=test` 下运行
- **强制**：`TEST_DATABASE_URL` 必须指向含 `_test` 的库
- **隔离**：每个测试使用独立事务，测试后自动回滚
- **迁移**：测试 session 开始前自动执行 `alembic upgrade head`

### 3.2 运行测试

```bash
cd /root/web_dev/backend
source .venv/bin/activate

# 运行全部测试
APP_ENV=test \
TEST_DATABASE_URL="postgresql://bz:***@127.0.0.1:5433/bz_stock_test" \
pytest tests/ -q

# 运行单个测试文件
APP_ENV=test \
TEST_DATABASE_URL="postgresql://bz:***@127.0.0.1:5433/bz_stock_test" \
pytest tests/test_auth.py -v

# 运行指定测试
APP_ENV=test \
TEST_DATABASE_URL="postgresql://bz:***@127.0.0.1:5433/bz_stock_test" \
pytest tests/test_auth.py::test_login_success -v
```

SELECT query, mean_exec_time, calls
FROM pg_stat_statements
ORDER BY mean_exec_time DESC LIMIT 10;

# 查看锁等待
SELECT pid, query, mode, granted
FROM pg_locks l JOIN pg_stat_activity a USING (pid)
WHERE NOT granted;

## 7. 代码规范

### 7.1 后端

- **Linter**：ruff（line-length=100）
- **Type Checker**：mypy（非 strict 模式）
- **格式化**：ruff format
- **命名**：snake_case（Python）/ PascalCase（类）
- **异常**：禁止吞异常（`except: pass`），必须补上下文后 re-raise
- **注释**：`# [业务模块] - 描述: xxx` 格式
- **自文档化优先**：能用函数名表达的不写注释

### 7.2 前端

- **Linter**：eslint
- **格式化**：prettier
- **类型**：TypeScript strict
- **样式**：JSX + SCSS（逐步移除 MUI props 样式，禁止新增）
- **toast**：使用全局封装

### 7.3 注释规范

新增注释时检查：
1. 是否可以通过优化函数命名替代注释（自文档化优先）
2. 若必须添加，使用统一格式：`# [业务模块] - 描述: xxx`

## 8. 文档生成

### 8.1 自动生成文档

```bash
cd /root/web_dev
DATABASE_URL="postgresql+psycopg://bz:***@127.0.0.1:5432/bz_stock" \
backend/.venv/bin/python tools/update_docs.py
```

生成：
- `docs/数据结构.md`：6 张 bar 表结构 + instruments 表 + DSA 历史回补表
- `docs/操作手册.md`：API 规格 + 调度任务 + 监控指标 + 故障排查

### 8.2 一致性检查（CI 用）

```bash
python tools/update_docs.py --check
# 不一致时返回非 0 退出码
```

### 8.3 文档维护规则

- **自动生成文档**：`数据结构.md` / `操作手册.md`，禁止手工编辑，只能改事实源与生成脚本
- **人工维护文档**：其余文档（`产品与业务规则.md` 等），由开发团队按版本更新
- **变更同步**：脚本参数 / 输出 / 表结构 / 语义变更后必须运行 `python tools/update_docs.py` 重建并提交 diff
