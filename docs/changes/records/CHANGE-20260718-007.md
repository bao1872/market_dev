## 2. 修改前理解

- **P0 阻断项（80 端口前端）**：
  - 生产入口固定 `80:80`，但 `frontend/Dockerfile` 误将构建产物 `COPY --from=builder /app/dist /usr/share`

- **CI 诊断**：
  - 25 个 Ruff 新诊断（W292/I001/F401/C401/F541/F841）跨 10+ 文件
  - 4 个 mypy 新错误（attr-defined/arg-type/assignment）
  - 4 个 Architecture Rules 违规（duplicate-plan-feature-list）
  - 前序会话本地检查 "All checks passed" 但 CI baseline 对比捕获了新诊断

## 10. S3.1：复权因子审计接入生产流水线（commit 7ea11e2 / ef0ff23 / c0f1bb7）

### 10.1 核心发现：auditor 日期类型 bug（审计从未真正工作过）

`FactorConsistencyAuditor._compare_factors` 存在严重的日期类型不一致 bug：

- **stored trade_date** 来自 DB `bars_daily` 表，dtype = `datetime64[s]`（pandas.Timestamp）
- **expected trade_date** 来自 `compute_expected_adj_factors`，dtype = `object`（datetime.date）
- `Timestamp('2023-01-03') != date(2023, 1, 3)` 恒为 True
- 导致**全部股票误报 `date_sequence_mismatch`**——审计从未真正工作过

**修复**：`pd.to_datetime(stored_sorted["trade_date"]).dt.date.tolist()` 统一归一化为 `date` 对象后再比较。

**验证**：新增 Case 7 自测覆盖日期类型不一致场景；auditor 40 个测试全绿。

### 10.2 生产闭环接入：bars_scheduler_service._audit_and_rebuild_factors

在 `bars_scheduler_service._process_all_instruments` 的日线阶段，于 `_rebuild_factors_if_needed` 之后、`_check_daily_coverage_and_trigger_dsa` 之前插入因子一致性审计步骤：

```
原始日线刷新 → 公司行为/factor重建 → [NEW] 因子一致性审计+串行重建 → 覆盖率门禁/DSA → 15m/60m刷新
```

**设计要点**：
- Phase 1: `FactorReconciliationTask.dry_run()` 全市场只读审计 → `ReconciliationPlan`
- Phase 2: 若 `needs_rebuild > 0`，`rebuild_batch()` 串行重建（每只股票独立事务）
- 写 FACTOR_AUDIT job_run_event（start + done，含 before/after hash 摘要）
- **软失败**：审计/重建异常不阻断 DSA（DSA 可基于旧因子 degraded 运行），但失败清单写入事件留下诊断痕迹
- `BatchResult` 新增 `factor_audit` 字段记录审计摘要

### 10.3 独立审计脚本：backend/scripts/run_factor_audit.py

支持：
- `--rebuild`：审计发现不一致时串行重建
- `--symbols 603629,603538`：仅审计指定股票
- `--output /tmp/report.json`：报告输出路径
- 输出 PROMPT.md S3.1 要求的真实数字：全市场审计总数 / mismatch 数 / 重建成功数 / 失败清单 / 样本股 before/after hash
