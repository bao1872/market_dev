# CHANGE-20260703-016

## 变更编号

CHANGE-20260703-016

## 任务名称

修复 worker_heartbeats 僵尸 running 记录清理机制

## 需求出处

生产只读审计（2026-07-04）发现 `worker_heartbeats` 表存在 stale/running 僵尸记录：

用户于 2026-07-03 明确要求：
"修复 worker_heartbeats 僵尸 running 记录，使后台 Worker 状态可观察性可信。
这是小范围修复，不是业务功能开发，不是重构。"

## 当前问题

1. `worker_heartbeats` 表中 `status='running'` 但 `heartbeat_at` 过旧的僵尸记录没有被标记为 stopped；
2. 容器被 SIGKILL（无 SIGTERM graceful shutdown）时，`_heartbeat_loop` 无法执行退出清理，记录永远停留 `status='running'`；
3. `_recovery_watchdog_loop` 只调用 `recover_stale_scheduler_job_runs`（清理 `scheduler_job_runs` 表），不清理 `worker_heartbeats` 表；
4. `system_overview_service._compute_base_fields` 的 `all_running_workers` 包含僵尸记录，导致 `worker_health` 误报 "degraded" 而非 "unknown"；
5. 生产审计证据：38 条 `status='running'`，仅 8 条活跃（age<120s），30 条僵尸（age 4333s ~ 153588s）。

## 当前设计理解

- `WorkerHeartbeat` ORM 模型位于 `backend/app/models/worker_heartbeat.py`，复合主键 `(worker_name, instance_id)`，`status` 字段合法值为 `running/idle/stopped`；
- `_heartbeat_loop`（`worker.py:60-132`）每 60s 更新 `heartbeat_at` + `status='running'`，graceful exit 时设 `status='stopped'`；
- `_recovery_watchdog_loop`（`worker.py:1250-1278`）每 60s 调用 `recover_stale_scheduler_job_runs` 恢复僵尸任务，但不清理 `worker_heartbeats`；
- `recover_stale_scheduler_job_runs`（`scheduler_job_run_recovery_service.py`）是 `scheduler_job_runs` 表的清理逻辑，与 `worker_heartbeats` 表职责不同，不合并；
- 不新增数据库 migration：`status` 字段已是 `String(32)`，`stopped` 是已有合法值，无需 enum 变更。

## 实际修改内容

### 代码修改

1. `backend/app/worker.py`：
   - 新增模块级常量 `STALE_HEARTBEAT_THRESHOLD_SECONDS = 600`（10 分钟，10 个心跳周期，远大于 60s 心跳间隔）
   - 新增 `from sqlalchemy.ext.asyncio import AsyncSession` 导入
   - 新增 `async def mark_stale_worker_heartbeats(db, now=None, threshold_seconds=600) -> int` 函数：
     - 原子 UPDATE `worker_heartbeats SET status='stopped' WHERE status='running' AND heartbeat_at < now - threshold`
     - 不 commit（由调用方控制事务，与 `recover_stale_scheduler_job_runs` 模式一致）
     - 不吞异常，返回被标记数量，使用 timezone-aware UTC
   - 修改 `_recovery_watchdog_loop`：在 `recover_stale_scheduler_job_runs` 调用后增加 `mark_stale_worker_heartbeats(db)` 调用，同事务内 commit

### 测试修改

2. `backend/tests/test_worker_heartbeat_stale_cleanup.py`（新增）：
   - `test_fresh_running_heartbeat_not_marked`：fresh running（age=30s）不被标记
   - `test_stale_running_heartbeat_marked_stopped`：stale running（age=1200s）被标记为 stopped
   - `test_non_running_heartbeat_not_processed`：已 stopped 的记录不被重复处理
   - `test_multiple_workers_batch_processing`：5 条记录（2 stale + 1 fresh + 1 stopped + 1 stale 同名不同实例）批量处理 3 条
   - `test_return_value_matches_marked_count`：返回值与实际标记数量一致
   - `test_threshold_boundary`：age=600（等于阈值）不被标记，age=601 被标记
   - `test_watchdog_calls_mark_stale_worker_heartbeats`：watchdog 集成测试，验证 stale heartbeat 被标记为 stopped
   - `test_idempotent_multiple_calls`：连续两次调用，第二次返回 0

## 修改前行为

- `worker_heartbeats` 表中 `status='running'` 但 `heartbeat_at` 过旧的僵尸记录永远保持 `running` 状态；
- 容器被 SIGKILL 时无 graceful exit 清理；
- `_recovery_watchdog_loop` 只清理 `scheduler_job_runs`，不清理 `worker_heartbeats`；
- 管理员看到的 Worker 状态包含 30 条假 "running" 记录；
- `system_overview_service` 的 `worker_health` 误报 "degraded" 而非 "unknown"。

## 修改后行为

- `_recovery_watchdog_loop` 每 60s 调用 `mark_stale_worker_heartbeats`，将 `status='running' AND heartbeat_at < now - 600s` 的记录标记为 `stopped`；
- 僵尸心跳被清理，管理员看到的 Worker 状态只包含活跃 worker；
- `system_overview_service` 的 `all_running_workers` 不再包含僵尸记录，`worker_health` 准确反映真实状态；
- 历史记录不删除，保留 `started_at`/`heartbeat_at`/`build_sha` 供审计。

## 阈值

- `STALE_HEARTBEAT_THRESHOLD_SECONDS = 600`（10 分钟）
- 依据：心跳间隔 60s，10 个心跳周期未刷新视为僵尸，远大于正常抖动，避免误杀活跃 worker
- 可通过环境变量 `STALE_HEARTBEAT_THRESHOLD_SECONDS` 覆盖

## 影响范围

- 仅影响 `worker_heartbeats` 表 `status` 字段，不影响 `scheduler_job_runs`、`message_deliveries`、`outbox` 等业务表
- 仅在 watchdog 循环中新增调用，不改变各 WORKER_TYPE 的启动方式
- 不改变业务任务调度时间
- 不改变 `scheduler_job_runs` 的既有恢复逻辑
