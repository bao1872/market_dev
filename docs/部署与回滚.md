# 部署与回滚

## 1. 环境准备

### 1.1 PostgreSQL（Docker 容器）

- **镜像**：`postgres:16`
- **服务名**：`postgres`（docker-compose.prod.yml 中的服务，应用容器通过服务名连接）
- **容器名**：`trading-postgres`
- **时区**：`TZ=Asia/Shanghai`
- **端口**：不映射到宿主机，仅容器网络内通过 `postgres:5432` 访问
- **数据卷**：`postgresdata`（持久化存储）
- **测试库容器**：`trading-postgres-test`（端口 5433，仍部署在物理机上，保持不变）

**容器初始化**（首次部署）：

PostgreSQL 容器通过环境变量自动创建用户与数据库，无需手动执行 SQL：
- `POSTGRES_USER`：业务用户（默认 `bz`），容器启动时自动创建
- `POSTGRES_PASSWORD`：业务用户密码（必填，通过 env 文件注入）
- `POSTGRES_DB`：默认数据库（默认 `bz_stock`），容器启动时自动创建

> 容器首次启动时会基于 `postgresdata` 数据卷判断是否初始化；已初始化的数据卷不会重复执行 `POSTGRES_USER` / `POSTGRES_DB` 逻辑。

### 1.2 外部 env 文件

生产环境 env 文件位置：`/etc/market-dev/market.env`

**必填变量**：
```bash
# PostgreSQL 容器初始化（docker-compose.prod.yml 中 postgres 服务读取）
POSTGRES_USER=bz
POSTGRES_PASSWORD=***
POSTGRES_DB=bz_stock

# 应用连接串（主机为 Docker 服务名 postgres）
DATABASE_URL=postgresql+psycopg://bz:***@postgres:5432/bz_stock
JWT_SECRET=<strong-random-secret>
SECRET_MASTER_KEY=<strong-random-secret>
APP_ENV=production
```

**可选变量**：
```bash
REDIS_URL=redis://redis:6379/0
JWT_ALGORITHM=HS256
JWT_ACCESS_TTL_SECONDS=3600
JWT_REFRESH_TTL_SECONDS=604800
FRONTEND_BASE_URL=http://frontend

```

**管理员飞书通知变量**（v9 新增，用于内测申请通知）：
```bash

ADMIN_FEISHU_SIGN_SECRET=xxxxxxxxxxxxxxxx
```
- `ADMIN_FEISHU_WEBHOOK_URL`：必填，管理员飞书群 Bot Webhook URL。
- `ADMIN_FEISHU_SIGN_SECRET`：可选，飞书 Webhook 签名校验密钥。

### 1.3 Docker Compose 文件

- **开发环境**：`docker-compose.yml`（仅 Redis 容器，PostgreSQL 由开发者本地配置）
- **生产环境**：`docker-compose.prod.yml`（包含全部服务，含 PostgreSQL 容器）

### 1.4 生产环境配置挂载（config.production.py）

生产环境通过 `CONFIG_FILE` 环境变量指定额外配置文件，并挂载宿主机上的 `/etc/market-dev/config.production.py`：

```bash
sudo mkdir -p /etc/market-dev
sudo tee /etc/market-dev/config.production.py <<'EOF'
# 生产环境专属配置，覆盖代码默认项
# 例如：日志级别、外部服务地址、业务阈值等
EOF
sudo chmod 600 /etc/market-dev/config.production.py
```

所有后端与 Worker 服务统一设置：

```yaml
environment:
  CONFIG_FILE: /app/app/config.production.py
volumes:
  - /etc/market-dev/config.production.py:/app/app/config.production.py:ro
```

> 若 `/etc/market-dev/config.production.py` 不存在，需先创建空文件或移除对应挂载，否则容器启动会因挂载失败而报错。

### 2.4 Dockerfile 与 deploy.sh 说明

**后端 Dockerfile**（`backend/Dockerfile`）已在 runtime 阶段包含 Alembic 配置与迁移脚本：
```dockerfile
COPY alembic.ini ./
COPY alembic ./alembic
```
确保容器内可直接执行 `alembic upgrade head` / `alembic current`。

**统一发布脚本**（`scripts/deploy.sh`）执行流程：
1. 输出部署前后磁盘与镜像状态。
2. 清理前端 TypeScript 增量缓存，避免本地缓存导致 Docker 构建"假通过"。
3. 构建 backend / frontend / worker-capture 镜像。
4. 在容器内执行 `alembic upgrade head`（通过 `docker compose run --rm backend alembic upgrade head`），禁止 APP_ENV=development 直连生产库。
5. `docker compose up -d --no-build --force-recreate --remove-orphans` 启动所有服务。

### 3.4 镜像复用与 --no-build 部署

若当前 `GIT_SHA` 对应的镜像已存在且无需重新构建（仅配置或 env 变更），可直接复用镜像：

```bash
cd /root/web_dev
GIT_SHA=$(git rev-parse --short HEAD)

# 校验镜像是否存在
if docker image inspect "market-dev-backend:${GIT_SHA}" >/dev/null 2>&1; then
  echo "镜像已存在，跳过构建"
  docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml up -d --no-build --force-recreate --remove-orphans
else
  echo "镜像不存在，执行构建"
  GIT_SHA=$(git rev-parse --short HEAD) \
  BUILD_TIME=$(date -u +"%Y-%m-%dT%H:%M:%SZ") \
  docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml build
  docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml up -d --force-recreate --remove-orphans
fi
```

> 日常发布不再默认执行 `docker image prune` / `docker builder prune`，避免误删相邻版本镜像或导致回滚时重新构建。清理操作仅在磁盘告警时按需执行（见第 7 节）。

### 4.1 代码回滚

```bash

# 查看最近 5 次提交

# 回滚到指定 commit
```

### 4.2 数据库迁移回滚

```bash
cd /root/web_dev

# 回滚一个版本
docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml run --rm backend alembic downgrade -1

# 回滚到指定版本

```

> **注意**：数据库回滚可能造成数据丢失，仅在迁移有 down 方法时可用。生产回滚前必须先备份。

### 4.3 服务回滚

```bash

GIT_SHA=$(git rev-parse --short HEAD) \
BUILD_TIME=$(date -u +"%Y-%m-%dT%H:%M:%SZ") \
docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml build

docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml up -d --force-recreate --remove-orphans
```

> 回滚时优先使用 `--no-build` 复用旧版本镜像；仅当镜像缺失时才重新构建，避免误删可用于回滚的相邻版本镜像。

### 4.4 数据库备份与恢复


## 7. 磁盘清理

### 7.1 镜像堆积原因

每次 `docker compose build` 会创建新镜像，旧镜像变为 `<none>` 标签（悬空镜像）但不会自动删除。多次部署后镜像和 build cache 会大量堆积，导致磁盘占用持续上升。

### 7.2 清理命令

```bash
# 清理悬空镜像（<none> 标签的旧镜像）
docker image prune -f

# 清理 build cache（构建缓存）
docker builder prune -f

# 查看磁盘占用
docker system df
df -h /
```

### 7.3 部署后自动清理

建议在部署脚本中加入自动清理：
```bash
docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml build
docker image prune -f  # 部署后立即清理悬空镜像
docker compose --env-file /etc/market-dev/market.env -f docker-compose.prod.yml up -d --force-recreate
```

### 7.4 定期清理

建议每周清理一次 build cache（crontab）：
```bash
# 每周日凌晨 3 点清理 build cache
0 3 * * 0 docker builder prune -f > /dev/null 2>&1
```

### 7.5 监控指标

- `/var/lib/docker` 目录大小：`du -sh /var/lib/docker`
- 可回收空间：`docker system df` 的 RECLAIMABLE 列
- 磁盘使用率：`df -h /`（目标 < 75%）

## 10. 权限架构 V2 迁移指引

本节给出权限架构 V2（Phase 2）配套的 Alembic 迁移顺序、部署门禁与重启评估。Phase 2 将会员体系从单表 `memberships` 迁移至 `plans` + `subscriptions` 两表模型，迁移不可逆且依赖 backfill 完成，必须严格按顺序执行。

### 10.1 Alembic 迁移顺序

事实源：`backend/alembic/versions/048_plans_table.py` + `049_subscriptions_table.py`

| 序号 | 迁移文件 | 动作 | 备注 |
|------|---------|------|------|
| 048 | `048_plans_table.py` | 创建 `plans` 表 + 预置 `observe_20` / `research_50` 套餐 | 套餐元数据，幂等 |
| 049 | `049_subscriptions_table.py` | 创建 `subscriptions` 表 + 迁移 `memberships` 数据 + 删除 `memberships` 表 | 数据迁移，不可回滚（down 方法仅删表，不恢复 memberships） |

执行顺序必须为 048 → 049，不可跳过 048（049 的外键依赖 plans 表）。

### 10.2 部署门禁

迁移执行前必须满足以下门禁，任一未通过则停止部署：

1. **trading-bars-backfill 必须完成**：进程退出码必须为 `exit=0`。Phase 2 的订阅状态计算依赖历史 K 线数据，backfill 未完成会导致订阅判定异常。
2. **迁移前快照**：049 含数据迁移（memberships → subscriptions），必须先 pg_dump 全量备份：
   ```bash
   docker compose -f docker-compose.prod.yml exec postgres pg_dump -U bz -d bz_stock -F c -f /backup/bz_stock_pre_phase2_$(date +%Y%m%d_%H%M%S).dump
   ```
3. **迁移后验证**：
   ```bash
   # 验证 plans 表预置数据（预期：observe_20 / research_50）
   docker compose -f docker-compose.prod.yml exec postgres psql -U bz -d bz_stock -c "SELECT plan_code FROM plans ORDER BY plan_code;"

   # 验证 subscriptions 表数据已从 memberships 迁移
   docker compose -f docker-compose.prod.yml exec postgres psql -U bz -d bz_stock -c "SELECT count(*) FROM subscriptions;"

   # 验证 memberships 表已删除（预期：未找到该表）
   docker compose -f docker-compose.prod.yml exec postgres psql -U bz -d bz_stock -c "\dt memberships"
   ```
