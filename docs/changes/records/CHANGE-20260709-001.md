# CHANGE-20260709-001

## 变更编号
CHANGE-20260709-001

## 任务名称
research feature matrix DB 主存储 + compute + writer + CLI

## 需求出处
用户要求把研究矩阵主存储从 parquet 改成数据库：
1. 不写 production stock_feature_snapshots；
2. 新增 research 专用轻量表 `research_feature_matrix_runs` + `research_feature_matrix_rows`；
3. rows 表用扁平字段，不存完整 JSON payload，不写大 JSONB，不建 GIN index；
4. registry 仍保留 dotted key（`causal.atr`），写 DB 时映射成 `causal_atr`；
5. 按月分批回补（`--month YYYY-MM`），每月一个 run；
6. 支持 `--dry-run` 估算 `expected_rows` / `estimated_db_size`；
7. 支持 `--resume` 幂等 upsert；
8. 禁止直接全量跑到当前，必须分阶段验证；
9. 硬阈值：磁盘 < 15GB / 单月 > 3GB / 失败率 > 5% 停止；
10. 不生成中间 parquet/CSV/coverage/截图/大日志/DB 备份。

## 修改前行为
- 上一 PR（CHANGE-054）只完成 registry 骨架 + CLI 骨架，登记 27 字段（causal 10 + confirmed_delay 4 + hindsight 6 + label 7）；
- CLI 默认 `--dry-run`，不写 DB，不写文件；`--output` 配合 sample scope 才允许输出 parquet；
- 文档 `06-research-feature-matrix.md` 第 1 节明确"研究矩阵不新增数据库表"；
- 历史 full snapshot 回补仍 BLOCKED（PR #41 126min > 120min 阈值）。

## 修改后行为

### 1. registry 扩展到 33 字段

`backend/app/research/feature_causality_registry.py`：
- causal 从 10 扩展到 16：新增 `causal.bb_percent_b` / `causal.bb_bandwidth_pct` / `causal.sqzmom_val` / `causal.sqzmom_delta_1` / `causal.active_swing_dir` / `causal.active_swing_high` / `causal.active_swing_low` / `causal.developing_swing_dir` / `causal.developing_swing_high` / `causal.developing_swing_low`（拆分原 `causal.bb` / `causal.sqzmom` / `causal.active_swing` / `causal.developing_swing` 为多个独立字段）；
- confirmed_delay 4、hindsight 6、label 7 不变；
- 合计 33 字段（causal 16 + confirmed_delay 4 + hindsight 6 + label 7）。

### 2. `FeatureSpec.db_column` 属性

新增 `db_column` property：把 dotted key 映射为 DB 下划线列名（`causal.atr` → `causal_atr`，`hindsight.dsa_finalized_segment` → `hindsight_dsa_finalized_segment`）。

### 3. 新增 ORM 模型

`backend/app/models/research_feature_matrix.py`：
- `ResearchFeatureMatrixRun`：16 列（id/run_key/month/start_date/end_date/status/instruments_count/trade_dates_count/rows_count/failed_count/duration_seconds/started_at/finished_at/metadata_json/created_at/updated_at）；
  - `run_key` 唯一约束 + month/status 2 个 btree 索引；
  - `metadata_json` 是 JSONB 但只放小摘要，不存完整 payload，不建 GIN 索引；
  - 状态机：`running` → `succeeded` / `failed`；
- `ResearchFeatureMatrixRow`：39 列（5 metadata + 33 feature + 1 created_at）；
  - 33 个 feature 列与 `feature_causality_registry.db_column()` 1:1 对应；
  - `(instrument_id, trade_date)` 唯一约束跨 run 幂等 upsert；
  - 3 个 btree 索引（trade_date / instrument_id / run_id），不给单个 feature 列建索引；
  - 扁平宽表设计，不存 JSON payload，不建 GIN 索引；
  - 所有 feature 列允许 NULL（warmup 期、未来 label 未到计算期等）。

### 4. 新增 migration 058

`backend/alembic/versions/058_research_feature_matrix.py`：

- 含唯一约束 + 索引；
- **本 PR 不应用 migration**：留到 PR merge 后统一应用，2 symbols/100 stocks/full market 验证延后。

### 5. 新增 `feature_computer` 计算模块

`backend/app/research/feature_computer.py`：
- `compute_all_features(bars)` 返回 DataFrame，index 为 trade_date，包含 33 个 feature 列；
- 与生产 `structural_factor_service` 的区别：生产只计算最后一根 bar 的 single-snapshot 值，研究矩阵计算 per-bar full series；
- 底层复用现有算法 SSOT（不重写公式）：`compute_atr` / `bollinger` / `compute_sqzmom_lb` / `_tv_pivots_confirmed` / `compute_dsa_history`；
- 4 命名空间函数：`compute_causal_rolling_features` / `compute_confirmed_delay_features` / `compute_hindsight_features` / `compute_label_features`。

### 6. 新增 `research_matrix_writer` 写入模块

`backend/app/research/research_matrix_writer.py`：
- 三道硬阈值纯函数：`check_disk_threshold`（< 15GB 停止）/ `check_month_size_threshold`（> 3GB 停止）/ `check_failure_rate`（> 5% 停止）；
- 月份解析：`resolve_month_range(month)` 用 `calendar.monthrange` 处理闰年；
- 单月大小估算：`estimate_month_size(instruments_count, trade_dates_count)` 返回 GB；
- monthly run 生命周期：`create_or_resume_run`（`run_key = f"{month}_{scope}"`，相同 run_key 返回已存在 run）/ `finalize_run`；
- 批量 upsert：`upsert_rows_batch(db, rows)` 使用 PostgreSQL `INSERT ... ON CONFLICT (instrument_id, trade_date) DO UPDATE` 幂等覆盖；自动分批 `UPSERT_BATCH_SIZE=1000`；
- 常量：`DISK_MIN_GB=15` / `MONTH_SIZE_MAX_GB=3.0` / `FAILURE_RATE_MAX=0.05` / `_BYTES_PER_ROW=2048`。

### 7. 重写 CLI 入口

`backend/scripts/research_feature_matrix_backfill.py`：
- `--month YYYY-MM`（推荐用法）与 `--start/--end` 互斥必填其一；
- 移除 `--output` / `--include-hindsight` / `--include-labels`（始终计算全部 33 字段）；
- 新增 `--resume` flag（幂等 upsert）；
- 新增 `--export-parquet PATH`（可选 debug 导出，仅 sample scope，不作为主存储）；
- 主流程：解析日期 → 磁盘阈值 → universe+trade_dates → 单月大小阈值 → dry-run 退出 → 创建/resume run → instrument-first 循环（每只股票 load bars 1次 → compute_all_features → 按月份 trade_date 切片 → upsert）→ 失败率检查 → finalize run → 可选 export parquet；
- `_WARMUP_DAYS = 400`（加载 start_date - 400 天 bars 作为 warmup，约 16 个月）；
- 每 100 只 instrument commit 一次；
- tqdm 进度条包装 instruments_iter。

## 风险
- migration 058 未应用：代码 merge 后但 migration 未跑时，CLI 运行会因表不存在失败；需在 merge后第一时间应用 migration；
- 单月回补实际耗时未验证（dry-run 只验证计划，不验证计算性能）：需要在 PR merge + migration 后跑阶段 B/C/D 才能确认；
- 磁盘阈值检查 `shutil.disk_usage("/")` 在容器内查的是容器根分区，生产部署需确认容器 `/` 与宿主机 `/` 的磁盘关系；
- asyncpg `ON CONFLICT DO UPDATE` 的 `result.rowcount` 在某些场景可能不区分 insert/update，writer 返回 `len(rows)` 而非 `result.rowcount`，幂等 upsert 的写入行数等于提交行数；
- 33 字段的 NULL 处理：warmup 期、未来 label 未到计算期、compute 失败等情况都可能产生 NULL，回测/分析使用时需注意。

## 遗留问题
