# CHANGE-20260708-053

## 变更编号
CHANGE-20260708-053

## 任务名称
feature_snapshot_backfill 轻量 profile-summary 性能诊断模式

## 需求出处
PR #41 生产验证显示 full scope backfill 运行 126 分钟，超过 120 分钟阈值，历史回补被 BLOCKED。需要轻量 profile-summary 模式定位瓶颈（查询/计算/upsert/commit/PG 等待），不重构公式，不跑历史回补。

## 修改前行为
`feature_snapshot_backfill.py` 无性能诊断能力。生产 backfill 运行 126 分钟但无法定位瓶颈在 load_bars / compute / upsert / commit 哪个环节，也无法评估多 worker 的扩展性。

## 修改后行为
1. 新增 `--profile-summary` CLI 参数（默认 False，不启用）
2. 新增 `ProfileCollector` 类：轻量收集器，record() 收集单步耗时，merge() 合并多 worker，compute_stats() 计算 total/avg/p50/p95，format_summary() 输出聚合统计
3. 单进程模式（`backfill_instrument_first`）：传入 profile 时对 load_bars_ms / compute_ms / upsert_ms / total_ms_per_instrument 计时
4. 多进程模式（`_worker_process_instruments` + `backfill_instrument_first_parallel`）：每个 chunk 创建独立 ProfileCollector 传给 worker，worker 返回 (stats, profile) 元组，主进程 merge 到主 profile
5. 失败路径也计入 compute_ms 耗时（compute 失败时记录，upsert 失败时不重复记录）
6. dry-run 模式不收集 timing（不进入主循环）
7. 每 50 instruments 输出进度摘要（load_avg/compute_avg/upsert_avg/total_avg）
8. 结束时 stdout 输出聚合统计：instruments_total / trade_dates_total / rows_new/skipped/failed / load_bars_ms / compute_ms / upsert_ms / total_ms_per_instrument（各含 total/avg/p50/p95）/ worker_count / estimated_full_day_time
9. 不改变 success/failed/skipped 统计口径
10. 不新增表，不写文件，不输出逐股票明细

## 风险
- profile-summary 增加少量 timing 开销（time.perf_counter + dict.append），每 instrument 约 4 次 record，对 126 分钟总耗时影响可忽略（<0.1%）
- 多进程模式下 ProfileCollector 通过 pickle 传回主进程，大样本时样本列表可能较大（每 5000 instruments × 4 步骤 = 20000 个 float，约 160KB，可接受）

## 遗留问题
- 待生产 profile 验证：workers=1/2/4 × limit=100 × sample 非 dry-run，定位 126 分钟瓶颈
- 子步骤计时（compute_primary_ms / compute_secondary_ms / temporal_ms）需要 service 层插桩，本次未实现（当前 compute_ms 为整体 compute 调用耗时，足够定位是否 compute 是瓶颈）
- 历史回补仍 BLOCKED，需 profile 结果决定是否进入 compute-once-extract 优化
