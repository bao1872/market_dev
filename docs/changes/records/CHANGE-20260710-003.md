# CHANGE-20260710-003: 盘后流水线 5 阶段时间线与进度可观测性修复

- 变更编号：CHANGE-20260710-003
- 任务名称：盘后流水线面板 5 阶段化、负耗时修复、进度事件落库、特征快照停滞判定与构建可追溯性校验
- 需求出处：2026-07-10 盘后任务（job_run_id=9c8eb7dc-2af3-48fd-a541-7b147105b6e7）页面长期停在 feature_snapshot 的根因排查 + 主管对展示口径的明确指令（面向用户仅 5 个真实阶段）

## 修改前行为
1. 面板按 `refreshing_daily → checking_coverage → creating_dsa → waiting_dsa_worker → quality_gate → feature_snapshot → publishing → watchlist_ready` 8 项展示；其中 `checking_coverage`/`creating_dsa`/`watchlist_ready` 在执行期从未作为 `orchestrator_status` 出现（或仅为门禁），却与真实执行阶段混排，用户无法对齐真实进度。
2. 进度回调 `_build_feature_snapshot_progress_callback` 先 `db.commit()`（写 metadata 进度）再 `append_event`（进度事件只 flush 不 commit），会话关闭后进度事件被丢弃 → 进度事件永远不进入 job_run_events，事件抽屉缺失进度条目。
3. `feature_snapshot` 被 `_update_orchestrator_status` 调用两次（步骤开始 + 写 run_id），产生两条相同的“开始生成特征快照”事件。
4. 状态切换事件无 `event_type` 语义标记，难以区分 started/progress。
5. `after_close_run` 摘要与 `AfterClosePipelineResponse` 均未暴露 `feature_snapshot_stalled`，前端无法提示“疑似停滞”。

## 修改后行为
1. **面向用户合并为 5 真实阶段**：`market_prep`(行情准备=refreshing_daily+checking_coverage+creating_dsa) → `dsa_compute`(DSA计算=waiting_dsa_worker) → `quality_gate`(质量校验) → `feature_snapshot`(特征快照) → `publishing`(发布结果)。内部 `orchestrator_status` 状态机与 job_run_events 全部保留不变，向后兼容；虚拟步骤经 `_PHASE_REP_FOR_STATUS` 归并到 `market_prep`。`watchlist_ready` 仅作为最终发布门禁，不作为执行步骤。
2. `_compute_step_states` 运行中阶段强制 `finished_at=None` 且 `duration=now-started`（严格为正）；已完成阶段 `finished>=started`，任何负耗时归零。阶段结束时间由下一阶段开始事件推导（兼容历史数据：旧事件无 event_type 时首事件时间即开始时间）。
3. 进度回调改为 **metadata 与进度事件同一次 commit**，进度事件正常持久化到 job_run_events。
4. 删除 `feature_snapshot` 重复“开始”事件：仅在创建 snapshot_run 后写一次带 `feature_snapshot_run_id` 的 started 事件（`after_close_orchestrator.py` 移除 run 创建前的那次调用）。
5. 状态切换事件 payload 增加 `event_type="started"`；feature_snapshot 进度事件 payload 增加 `event_type="progress"`，便于聚合与排查。
6. 新增纯函数 `_compute_feature_snapshot_stalled`：编排处于 `feature_snapshot` 且心跳新鲜，但 `feature_snapshot_progress.updated_at` 距今超过 `_FEATURE_SNAPSHOT_STALL_SECONDS=300s` 时返回 True；`after_close_run_summary.feature_snapshot_stalled` 与顶层 `feature_snapshot_stalled` 均暴露该标志（已补入 `AfterCloseRunSummary` schema，避免 FastAPI 丢字段），前端提示“疑似停滞”（不替代 blocked 心跳超时判定）。
7. 新增 `tools/check_build_traceability.py`：只读校验构建可追溯性——`GIT_SHA` 环境变量与 HEAD 一致、运行态 `worker_heartbeat.build_sha`（只读 DB）与 HEAD 一致、运行镜像 tag 非 `unknown`/`<none>`；不一致/未知 FAIL（SKIP 仅在 env 不可用），exit 0 表示 PASS/SKIP。本轮仅检测，不重建/重启。
