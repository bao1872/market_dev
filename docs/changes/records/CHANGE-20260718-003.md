# CHANGE-20260718-003: 磁盘和构建性能优化（Dockerfile ARG 重排 + BuildKit cache mount + 日志轮转 + cleanup 策略）

## 变更编号
CHANGE-20260718-003

## 任务名称
Docker 构建性能与磁盘治理：版本 ARG 移到依赖层之后避免缓存失效；启用 BuildKit pip/npm cache mount；基础镜像 digest 固定；移除 PIP_NO_CACHE_DIR；docker-compose json-file 日志轮转；daemon.json 日志默认 + builder gc；cleanup 脚本改为"当前 + 1 rollback"保留策略。

## 需求出处
`ref/PROMPT.md` 第五章「磁盘和构建性能」：版本 ARG 移到依赖层之后，只让源码/build 和最终 metadata 层失效；启用 BuildKit，pip/npm 使用 cache mount；固定基础镜像 digest，默认不 --pull；配置 Docker json-file 日志轮转 max-size/max-file；cleanup 脚本按"当前 + 1 rollback"保留，不等 7 天才处理同日多次构建产生的无引用 SHA 标签和 cache；完成后做一次冷构建和一次仅改代码的热构建，记录各层 cache hit、下载次数、耗时和磁盘增量。

## 修改前行为

### 问题 1: 版本 ARG 在依赖层之前，导致每次部署依赖层缓存失效
- `backend/Dockerfile`：`ARG GIT_SHA` / `ARG BUILD_TIME` 在 `pip install` 之前声明，GIT_SHA 变化使 pip install 层失效。
- `backend/Dockerfile` runtime 阶段：`ARG GIT_SHA` 在 `apt-get install nodejs` 之前声明，GIT_SHA 变化使 apt-get 层失效（每次部署重新下载 nodejs）。
- `frontend/Dockerfile`：`ARG GIT_SHA` 在 `npm ci` 之后但 `COPY . .` 之前，npm ci 层不受影响但 npm run build 受 ARG 影响（可接受）。
- `backend/Dockerfile.capture`：`ARG GIT_SHA` 在 `pip install` 之前，同样导致依赖层失效。

### 问题 2: PIP_NO_CACHE_DIR=1 禁用 pip 下载缓存
- `backend/Dockerfile` 和 `Dockerfile.capture` 设置 `ENV PIP_NO_CACHE_DIR=1`，每次构建都重新下载所有 pip 包（pandas/numpy 等大包）。

### 问题 3: 基础镜像用 tag 而非 digest，可能被意外更新
- `python:3.11-slim`、`node:20-alpine`、`nginx:alpine` 均用 tag，registry 镜像更新时构建结果不可复现。

### 问题 4: 无 Docker json-file 日志轮转
- `docker-compose.prod.yml` 所有服务无 `logging` 配置，容器日志无限增长可能占满根盘。
- `/etc/docker/daemon.json` 无 `log-driver` / `log-opts` 默认配置。

### 问题 5: cleanup 脚本保留策略过宽
- `scripts/cleanup-docker.sh` 保留最近 3 个 + 7 天过滤，同日多次构建产生的无引用 SHA 标签和 build cache 不被及时清理。
- `docker image prune` / `docker builder prune` 带 `until=7d` 过滤，同日累积不被处理。

### 问题 6: Makefile docker-build 未启用 BuildKit、未传递依赖锁标识
- `Makefile` `docker-build` target 直接 `docker compose build`，无 `DOCKER_BUILDKIT=1`，无 `PYPROJECT_LOCK_HASH`。

## 修改后行为

### 一、版本 ARG 移到依赖层之后（backend Dockerfile）
- **builder 阶段**：`ARG PYPROJECT_LOCK_HASH` 声明在 `COPY pyproject.toml` 之前（仅作 LABEL 审计，不影响 pip install 缓存键）；`pip install` 层仅由 `COPY pyproject.toml` 触发失效。
- **runtime 阶段**（关键修复）：`RUN apt-get install nodejs` → `COPY --from=builder /opt/venv` → `COPY alembic/scripts` → `COPY app` → **`ARG GIT_SHA/BUILD_TIME/PYPROJECT_LOCK_HASH`** → `ENV` → `LABEL`。
  - apt-get 层在 ARG 之前：GIT_SHA 变化不使 apt-get 缓存失效。
  - venv COPY 在 ARG 之前：BuildKit 按 venv 内容哈希缓存，pip install 不变则 CACHED。
  - 源码 COPY 在 ARG 之前：版本变化不使源码 COPY 层失效（仅源码变化触发）。
  - ENV/LABEL 在 ARG 之后：版本变化仅使 metadata 层失效（极低成本）。

### 二、BuildKit cache mount（pip / npm）
- `backend/Dockerfile`：`RUN --mount=type=cache,target=/root/.cache/pip pip install .` —— pip 下载缓存跨构建持久化，pyproject.toml 变化时只需下载增量包。
- `backend/Dockerfile.capture`：同上。
- `frontend/Dockerfile`：`RUN --mount=type=cache,target=/root/.npm npm ci` —— npm 下载缓存跨构建持久化。
- 所有 Dockerfile 添加 `# syntax=docker/dockerfile:1.4` directive 启用 BuildKit 前端。
- 移除 `ENV PIP_NO_CACHE_DIR=1`（改用 cache mount）。

### 三、基础镜像 digest 固定
- `python:3.11-slim@sha256:e031123e3d85762b141ad1cbc56452ba69c6e722ebf2f042cc0dc86c47c0d8b3`
- `node:20-alpine@sha256:fb4cd12c85ee03686f6af5362a0b0d56d50c58a04632e6c0fb8363f609372293`
- `nginx:alpine@sha256:54f2a904c251d5a34adf545a72d32515a15e08418dae0266e23be2e18c66fefa`
- `Dockerfile.capture` 保留 `mcr.microsoft.com/playwright/python:v1.45.0-jammy` tag（非本地镜像，维护时显式 --pull 更新并固定 digest）。
- 构建默认不 `--pull`，仅基础镜像维护时显式更新。

### 四、Docker json-file 日志轮转
- `docker-compose.prod.yml`：新增 `x-logging: &default-logging` YAML 锚点（`driver: json-file`，`max-size: "50m"`，`max-file: "5"`），14 个服务全部引用 `logging: *default-logging`，单容器日志上限 250MB。
- `/etc/docker/daemon.json`：新增 `log-driver: json-file` + `log-opts`（同 compose）+ `builder.gc`（`enabled: true`，`defaultKeepStorage: 20GB`）。

### 五、cleanup 脚本"当前 + 1 rollback"策略
- `scripts/cleanup-docker.sh` 重写：
  - `KEEP_VERSIONS=2`（当前 + 1 rollback）。
  - `docker image prune -f` / `docker builder prune -f` 移除 `until=7d` 过滤，同日立即处理。
  - `PROTECTED_REPO_PREFIXES` 保护基础镜像（python/node/nginx/postgres/redis/playwright），绝不删除。
  - 对 `market-dev-{backend,frontend,capture}` 按 CreatedAt 倒序保留前 `KEEP_VERSIONS` 个，其余 rmi。
  - rmi 前检查 `docker ps` 是否有运行容器使用该镜像，在用则跳过。
  - 禁止 `docker system prune -a` / volume prune / 删除生产数据。

### 六、Makefile 启用 BuildKit + 传递依赖锁标识
- `docker-build` target：
  - 前缀 `DOCKER_BUILDKIT=1`（确保旧版 docker 也能识别 BuildKit）。
  - 新增 `PYPROJECT_LOCK_HASH=$(sha256sum backend/pyproject.toml | cut -d' ' -f1)`，写入镜像 LABEL 供审计。
- `docker-compose.prod.yml` backend build args 新增 `PYPROJECT_LOCK_HASH: ${PYPROJECT_LOCK_HASH:-unspecified}`。

## 测试与验证

### 冷构建（--no-cache，从零建立缓存）
- **backend 冷构建**：
  - builder 阶段 pip install：85.9s（清华 PyPI 镜像 + BuildKit cache mount）。
  - runtime 阶段 apt-get nodejs：1253s（debian.org 网络慢，非 Dockerfile 问题）。
  - 总计 ~22min，磁盘 +3G（镜像 + build cache 1.258GB）。
- **frontend 冷构建**：
  - npm ci：4.4s（npmmirror + BuildKit cache mount）。
  - npm run build：15.7s（tsc + vite build）。
  - 总计 22s。

### 热构建验证（验证 ARG 重排修复）
- **第一次热构建**（发现 bug）：runtime 阶段 ARG 在 apt-get 之前 → GIT_SHA 变化使 apt-get 层失效 → 修复：ARG 移到所有 COPY 之后。
- **修复后验证**：
  - builder 阶段全 CACHED（FROM/WORKDIR/COPY pyproject.toml/venv/pip install 均命中缓存，pip install 0s vs 冷构建 85.9s）。
  - runtime 阶段 apt-get 需重建（ARG 位置变化导致缓存键改变，一次性成本）。
- **第二次热构建**（不同 GIT_SHA，相同源码）：apt-get 层 CACHED ✓（ARG 在 apt-get 之后，版本变化不影响 apt-get 缓存键）。

### 验证结论
- pip install 层跨源码变化 CACHED ✓（pyproject.toml 不变即缓存）。
- apt-get 层跨版本变化 CACHED ✓（ARG 在 apt-get 之后，GIT_SHA 变化不失效 apt-get 缓存）。
- npm ci 层跨源码变化 CACHED ✓（package*.json 不变即缓存）。
- 热构建不重跑 pip/npm 依赖安装 ✓。
