## 2. 修改前理解

- `StrategyBatchService.publish_run` 显式允许 `partial_failed` 状态发布，与自动发布门禁 `_check_quality_gates` 不一致。
- `MonitorBatchService` 直接调用 `bar_repository.get_bars` / `get_recent_bars`，未复用 `MarketDataAggregationService`，导致 monitor 链路缺少统一缓存、盘中 partial bar 聚合、数据源诊断和尾部补齐。
- `monitor_batch_service.py` 第 71 行注释仍写 `# 3600`，但 `NODE_CLUSTER_LOW_BARS` 已按 250×16 修正为 4000。
- `.github/workflows/ci.yml` 中 `ruff` 和 `type-check` job 设置 `continue-on-error: true`，失败不阻断 workflow。

## 3. 修改目标

- 统一 DSA 发布门禁：`publish_run` 仅允许 `completed` 状态进入 `published`，`partial_failed` 即使 `succeeded_count > 0` 也拒绝。
- monitor 行情统一走 `MarketDataAggregationService`，保持 Node Cluster 输入根数不变（1d=250、15m=4000、1m=2）。
- 修正 `monitor_batch_service.py` 中 `_15MIN_LOOKBACK_BARS` 的注释为 `# 4000 = 250 * 16`。
- CI 中 `ruff` / `type-check` 改为阻断（移除 `continue-on-error: true`）。
- 同步更新测试与当前设计文档。

## 5. 修改前行为

- admin 手动 `publish_run` 可发布 `partial_failed` 状态的运行，只要 `succeeded_count > 0`。
- monitor 链路直接查 DB 行情，不经过统一聚合服务，缺少缓存、partial bar 补齐和数据源诊断。
- `_15MIN_LOOKBACK_BARS` 注释写 `# 3600`，与 `NODE_CLUSTER_LOW_BARS=4000` 不符。
- CI 中 ruff / type-check 失败不阻断后续 job。

## 6. 修改后行为

- `publish_run` 仅 `completed` 可发布；`partial_failed` 显式拒绝，错误信息明确提示“仅 completed 可发布”。
- monitor 链路统一通过 `MarketDataAggregationService().get_bars` 获取行情，返回 DataFrame 再 `tail(N)` 保留 Node Cluster 所需根数。
- `_15MIN_LOOKBACK_BARS` 注释更新为 `# 4000 = 250 * 16`，与 `indicator_contract` 唯一真源一致。
- CI 中 `Ruff Changed Files` 阻断 PR 中新增或修改的 Python 文件，必须零错误；`Ruff Full Repository Baseline` 非阻断但会失败于基线回归；`type-check` 仍阻断。
- 新增 `tools/quality_baselines/ruff.json`，记录全仓库历史债务 922 个错误，任何 PR 不得使其增加。
