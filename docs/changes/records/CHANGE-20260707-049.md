# CHANGE-20260707-049: Backfill Multiprocessing 优化

## 变更编号
CHANGE-20260707-049

## 任务名称
Feature Snapshot Backfill multiprocessing 并行优化

## 需求出处
PR #39 部署后 500-instrument 小样本验证耗时 28 分钟，估算全市场 5293 instruments × 121 trade_dates 约 6.6 小时。
用户要求评估提效方法并实施 multiprocessing 优化（用户选择 "A. 仅 multiprocessing (推荐)"，不做 compute-once-extract 重构）。

## 修改前行为
1. `feature_snapshot_backfill.py` 只支持单进程模式（`backfill_instrument_first`），CPU 利用率低；
2. 单进程 commit 策略：整个 backfill 完成后一次性 commit（`main()` 控制），被 kill 中断会丢失所有未提交进度；
3. worker 循环中 `load_instrument_bars` 失败时只 rollback + 日志，不更新 `per_date_stats`，导致 `failure_rate` 计算不准确（load 失败的 instrument 不计入 failed）。

## 修改后行为
1. 新增 `--workers N` 参数（默认 1 单进程，>1 启用 multiprocessing）；
2. 新增 `_worker_process_instruments()`：top-level 可 pickle 的 worker 函数，每个 worker 进程独立创建 `async_engine` + `async_sessionmaker`，per-instrument commit（被 kill 不丢已完成，resume 可续）；
3. 新增 `backfill_instrument_first_parallel()`：主进程创建/finalize run records，通过 `ProcessPoolExecutor` + `loop.run_in_executor` 分发 instrument chunks 到 worker 进程，`asyncio.as_completed` 收集结果；
4. worker 循环重构为三阶段：1) load_bars（失败 → 所有 trade_dates 标 failed + continue）；2) 逐 trade_date compute + upsert；3) per-instrument commit（失败 → rollback + 日志）；
5. `main()` 根据 `args.workers > 1` 分支：并行模式从 `get_settings().database_url` 获取 db_url 传给 worker，单进程模式保持不变。

## 风险
1. **multiprocessing 隔离**：每个 worker 进程独立 `async_engine`，不能共享主进程连接池。`ProcessPoolExecutor` 需要 worker 函数可 pickle（`_worker_process_instruments` 为 top-level 函数，已验证）；
2. **per-instrument commit 性能**：相比单次 commit 多了 N 次 commit 开销，但换来了 resume 安全性（被 kill 不丢已完成）。对 5000+ instruments 的全量回补，per-instrument commit 的额外开销可接受；
3. **生产验证未执行**：本 PR 只在 test DB 验证单元测试，生产 dry-run + 小批量验证需部署后执行。

## 遗留问题
1. **生产 multiprocessing 验证**：需部署后执行 `--workers 4 --dry-run` + 小批量 `--symbols` 验证；
2. **全量回补策略**：multiprocessing 部署后需评估实际提速倍数，再决定历史回补分批策略；
3. **compute-once-extract 优化**：本次不做（70% 因子非因果，需 truncate per trade_date），未来可探索对因果因子（ATR/BB/volatility）的预计算优化。
