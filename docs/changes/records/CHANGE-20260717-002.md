## 修改前行为

### Bug #1: 行情读取出口分散
- 多个业务模块（indicator_service、strategy_batch、feature_snapshot、chart_bars、structural_factor、capture、monitor）各自调用 `bar_repository` 私有查询（`_query_*`/`_get_adj_factor_df`）或旧 `bar_repository.get_bars`，绕过统一出口。
- 复权在多处分散执行（`apply_adj_factor_to_bars`/`apply_adj_factor` 被业务层直接调用），存在二次复权风险。

### Bug #2: adj_factor 列信任 Bug（核心缺陷）
- pytdx hybrid bar（15:00:00 合成日线）和 15m/60m/1m 行内自带 `adj_factor=1.0`（错误值）。
- 旧 `_apply_adj_factor_core` 优先使用 bar 自带 `adj_factor` 列而非权威因子序列 `merge_asof` 结果 `_adj`：
  ```python
  if "adj_factor" in merged.columns:
      ratio = merged["adj_factor"] / denominator_factor  # 用 bar 自带列（pytdx=1.0，错误）
  ```
- 后果：除权日前后价格断层（qfq 未调整）。

### Bug #3: 复权无 point-in-time 语义
- 历史回算（盘后 snapshot、DSA）与当前页面使用同一因子序列（含未来除权事件），存在未来泄漏。

### Bug #4: 日内/周/月复权口径不统一
- 15m/60m/1m 信任行内旧 `adj_factor` 作为复权真源；周/月未明确"日线完成复权后再聚合"顺序。

### Bug #5: factor rebuild 只更新最近 5 根
- 公司行为因子更新时只刷新最新 5 根 bar 的 `adj_factor`，不重建完整历史因子序列。

### Bug #6: 因子失败用 1.0 伪装成功
- factor rebuild 失败返回 1.0 伪装成功，下游无法感知降级。

## 修改后行为

### 一、MDAS 行情读取唯一出口（Bug #1）
1. **`MarketDataAggregationService` (MDAS)** 为行情读取 + 复权应用 + 周/月聚合的唯一出口。
2. **职责边界**：
   - MDAS：读取 raw bars（经 repository 私有 `_query_*`）→ 调用 `AdjustmentFactorService` 获取权威因子序列 → 应用复权（仅一次）→ 周/月"日线完成复权后经 `kline_aggregator` 聚合" → 返回 bars + 诊断字段。
   - `bar_repository`：仅负责 raw OHLCV / 公司行为因子的 DB 读写和上游拉取，不负责复权应用。
   - `adj_factor`（计算模块）：纯计算，由 AdjustmentFactorService/MDAS 包装调用，业务层禁止直接导入。
   - `kline_aggregator`：周/月聚合出口，仅 MDAS 导入。
3. **业务层收口**：indicator_service、strategy_batch、feature_snapshot、chart_bars、structural_factor、capture、monitor、bars/indicators API 全部改走 MDAS，删除手工查询和二次复权。

### 二、adj_factor 列信任 Bug 修复（Bug #2）
`backend/app/services/adj_factor.py::_apply_adj_factor_core`：
- **修复前**：`if "adj_factor" in merged.columns: ratio = merged["adj_factor"] / denominator_factor`（用 bar 自带列，pytdx=1.0 错误）。
- **修复后**：始终使用权威因子序列 `merge_asof` 结果 `_adj`：
  ```python
  # 始终使用权威因子序列（merge_asof 的 _adj），不信任 bar 自带 adj_factor 列。
  # pytdx hybrid bar / 15m/60m/1m 行内旧 adj_factor 不可信（可能为 1.0），
  # 必须由权威日线因子覆盖（CHANGE-20260717-002 硬规则）。
  missing_count = int(merged["_adj"].isna().sum())
  merged["_adj"] = merged["_adj"].fillna(denominator_factor)
  ratio = merged["_adj"] / denominator_factor
  ```
- bars_df 自带 `adj_factor` 列仅作兼容保留，不参与复权计算。

### 三、adjustment_as_of point-in-time 复权（Bug #3）
1. **公式**：`qfq_price = raw_price × factor(bar_date) / factor(as_of)`
2. **`factor(as_of)`**：截至 `as_of` 的最近交易日因子（ffill），取**截断因子序列**（`trade_date <= as_of`）最后一个因子。
3. **`factor(bar_date)`**：截断序列中 `<= bar_date` 的最后一个因子（ffill）。
4. **无未来泄漏**：as_of 之后的除权事件不参与计算。MDAS 内部调用 `AdjustmentFactorService.get_factor_series(as_of=fetch_as_of)` 截断因子序列（`include_realtime=False` 时 `fetch_as_of=adjustment_as_of`）。
5. **当前页面**：`adjustment_as_of` 锚定请求业务日（None=最新因子，向后兼容）。
6. **盘后/历史回算**：`adjustment_as_of=trade_date`，只使用截至该日可知的公司行为和因子。
7. **adj=none 时**：`adj_factor_hash` 为空，不应用复权。

### 四、日内/周/月复权口径统一（Bug #4）
1. **日内（15m/60m/1m）**：同一交易日内所有 bar 映射到**同一权威日线因子**（`merge_asof` 按 trade_date 对齐）；禁止信任行内旧 `adj_factor`。
2. **周/月**：MDAS 先取 raw 日线 → 应用 qfq 复权（仅一次）→ 经 `kline_aggregator.convert_kline_frequency` 聚合；禁止在 raw 上聚合后再复权。

### 五、factor rebuild 完整重建（Bug #5）
1. **`AdjustmentFactorService.rebuild_factor_series`**：从**最早受影响日期**重新计算该股票完整日线 factor 序列并原子 upsert；禁止只更新最近 5 根。
2. **fingerprint**：`detect_company_action_change` → `earliest_affected` → rebuild → 成功后更新 fingerprint；失败回滚 fingerprint，可重试。
3. **缓存精确失效**：rebuild 成功后精确失效该股票的 MDAS / indicator 缓存（按 instrument_id + 算法版本隔离）。
4. **失败不伪装**（Bug #6）：rebuild 失败不得用 1.0 伪装成功，返回 degraded 状态和原因。

### 六、盘后顺序
`BarsScheduler` 顺序：原始日线刷新 → 公司行为/factor 重建（成功）→ 覆盖率门禁 / DSA → snapshot 发布。因子未完成时不得创建 DSA 或发布 snapshot。

### 七、MDAS v2 请求契约
1. **请求参数**：`timeframe`、`adj`、`include_realtime`、`completed_only`、`start_date`/`end_date`、`limit`、`warmup_bars`、`adjustment_as_of`。
2. **返回字段**：`bars` + `market_data_contract_version`、`source_bar_hash`、`adj_factor_hash`、`adjustment_as_of`、`completed_through`、`data_source`/`degraded`/`degraded_reason`。
3. **缓存键**：包含全部参数和算法版本，true/false 状态隔离。
4. **跨调用方一致性**：同一股票/周期/截止日下，`/bars`、indicator/SMC、strategy_batch、feature_snapshot 的时间序列、OHLC、`source_bar_hash`、`adj_factor_hash` 必须一致。

### 八、feature_snapshot schema v2
1. `feature_snapshot_service`/after-close 调用 MDAS，显式 `include_realtime=False, end_date=trade_date, adjustment_as_of=trade_date`。
2. 保存 `source_bar_hash`/`adj_factor_hash`/`contract_version`/`completed_through`/`adjustment_as_of` 到 run metadata。
3. 输入语义变化影响持久化结果时，snapshot schema version 递增，禁止新旧语义混用/覆盖。
4. 新增 alembic 迁移 063↔064（schema v2 落库），需测试库可逆验证。
