# CHANGE-20260710-002: 恢复飞书盘中截图 1d 业务契约，分离截图实时性与监控计算口径

## 变更编号

CHANGE-20260710-002

## 任务名称

修复 PR #65 后的业务语义偏差：飞书盘中截图错误使用 15m，且 docs/current 把错误 15m 写成验收、monitor 计算输入被截图修复污染。恢复业务规则：盘中监控触发仍基于最新已完成 1m bar；飞书盘中截图默认展示 1d；实时性由 Capture Snapshot 的 1d + include_realtime=True partial daily 保证；截图修复不得改变 watchlist_monitor 计算口径。

## 需求出处

主管指令（2026-07-10）：PR #65 merge 后发现飞书截图业务默认周期被误写为 15m，且 `monitor_batch_service` 的 `bars_daily`/`bars_15min` 计算输入被设为 `include_realtime=True`（为截图实时性污染事件计算口径）。要求恢复保守规则，并修正 docs/current / AGENTS.md / docs/maps 中把 15m 写成飞书业务默认/验收的错误。

## 修改前行为

- `stock_detail_feishu_service.send_stock_detail_to_feishu` 的 capture_payload 使用 `timeframe="15m"`；
- `monitor_batch_service._send_chart_images_via_outbox` 的 capture_payload 使用 `timeframe="15m"`；
- `monitor_batch_service._process_instrument_evaluation` 计算输入 `bars_daily` / `bars_15min` 均 `include_realtime=True`（注释称“供 current_price 与截图实时数据源”），使事件计算口径受截图实时性需求影响；
- `docs/current/05-testing-acceptance.md` 将“capture_payload 含 `timeframe=15m`”写成验收；`docs/current/03-jobs-integrations-operations.md` 调度语义写成“日线/15m 输入使用 `include_realtime=True`（供 current_price 与截图实时数据源）”。

## 修改后行为

- 新增常量 `FEISHU_CAPTURE_TIMEFRAME = "1d"`（`app/constants/capture.py`），两处飞书截图 capture_payload 统一使用该常量；
- 盘中监控触发仍只基于最新已完成 1m bar（`include_realtime=True` 且剔除最后一根未完成 bar），`source_bar_time` 来自最新已完成 1m；
- `monitor_batch_service` 计算输入 `bars_daily` / `bars_15min` 恢复 `include_realtime=False`（仅已完成 bar），watchlist_monitor 事件计算口径不再被截图实时性污染；current_price 实时展示由 card 的 `change_pct`（pytdx live）与 capture snapshot 1d partial daily 负责，与监控计算链路分离；
- Capture Snapshot API 继续支持多周期（1d/15m/1h/...），`include_realtime=True` 与周期透传硬规则不变——API 能力不等于飞书业务默认 15m；
- docs/current / AGENTS.md / docs/maps 已纠正：飞书盘中截图业务默认 1d、监控 daily/15m 计算输入非实时、15m 仅作为 API 能力或策略明确声明的辅助上下文。
