# 定时任务与运行手册

## 1. 任务总览

事实源：`backend/app/worker.py`

| 任务名 | WORKER_TYPE | 触发时间（Asia/Shanghai） | 执行间隔 | 依赖 |
|--------|-------------|--------------------------|----------|------|
| 行情刷新 | `bars_scheduler` | 每日 16:00 | Cron | 交易日 + pytdx |
| 选股兜底 | `strategy_scheduler` | 每日 18:30 | Cron | 交易日 + 行情数据就绪 |
| 日历刷新 | `calendar_scheduler` | 每日 02:00 | Cron | pytdx |
| 盘中监控 | `monitor_scheduler` | 交易日 9:30-11:30 / 13:00-15:00 | 30s 循环 | 交易日 + 自选股 |
| 策略批量 | `strategy_batch` | 无定时，轮询 | 5s | queued 状态的 run |
| Outbox Relay | `outbox` | 无定时，轮询 | 5s | pending 状态的 outbox |
| 投递 Worker | `delivery` | 无定时，轮询 | 5s | pending 状态的 MessageDelivery |
| 盘后编排 | `after_close_orchestrator` | 无定时，轮询 | 5s | queued 状态的 after_close 任务 |
| 看门狗 | `watchdog` | 无定时，循环 | 60s | 僵尸 running 任务（lease 过期或 heartbeat 超时） |

## 2. 行情刷新任务（bars_scheduler）

### 2.1 调度配置

- **触发时间**：每日 16:00（含非交易日，由内部交易日历判断是否执行）
- **Cron 表达式**：`day_of_week="mon-sun", hour=16, minute=0`，`timezone=Asia/Shanghai`
- **Job ID**：`bars_refresh_daily`
- **拉取方式**：串行（pytdx 不支持并发），通过 `asyncio.to_thread` 桥接
- **失败重试**：最多 3 次，间隔 5 秒

### 2.2 拉取配置

事实源：`backend/app/services/bars_scheduler_service.py`

**每日增量（DAILY_COUNTS，约 1.8 小时）**：

| 周期 | count |
|------|-------|
| d | 5 |
| 15m | 50 |
| 60m | 10 |

**历史回补（BACKFILL_COUNTS，约 11.1 小时）**：

| 周期 | count |
|------|-------|
| d | 500 |
| 15m | 15000 |
| 60m | 4000 |

### 2.3 执行流程

```
1. 交易日判断（is_trading_day_async）
2. 创建 SchedulerJobRun（status=running）
3. 后台心跳任务（每 30s 更新 heartbeat_at / lease_expires_at）
4. 分阶段处理：
   a. 日线优先（refresh_daily_bars）
   b. 覆盖率检查 ≥90%
   c. 自动触发 DSA（事件驱动）
   d. 15min（refresh_15min_bars）
   e. 60min（refresh_60min_bars）
5. 完成后更新 SchedulerJobRun（status=succeeded/failed）
6. 记录 strategy_run_id 到 metadata_json
```

### 2.4 状态检查


## 3. 选股兜底任务（strategy_scheduler）

### 3.1 调度配置

- **触发时间**：每日 18:30（比 bars 16:00 晚 2.5 小时，作为兜底）
- **Cron 表达式**：`day_of_week="mon-sun", hour=18, minute=30`
- **Job ID**：`strategy_run_daily`

### 3.2 执行流程

```
1. 交易日判断
2. 创建 SchedulerJobRun（status=running）
3. 查询 production 环境 + 参与调度 + 有 released 版本的 selector 策略
4. 逐个调用 StrategyBatchService.create_batch_run（幂等）
5. 每创建 5 个更新一次心跳
6. 完成后按 succeeded/failed 计数映射最终状态：
   - failed == 0 → succeeded
   - succeeded > 0 → partial_failed
   - 否则 → failed
```

### 3.3 兜底说明

- 18:30 触发作为兜底，日线触发优先（bars 完成后自动触发 DSA）
- `create_batch_run` 内部基于 `(version, date, run_type)` 与 `attempt_no` 幂等
- 数据就绪检查：覆盖率 < 90% 时阻断 DSA 执行

## 4. 日历刷新任务（calendar_scheduler）

### 4.1 调度配置

- **触发时间**：每日 02:00
- **Cron 表达式**：`hour=2, minute=0`
- **Job ID**：`calendar_scheduler`

### 4.2 执行流程

```
1. 创建 SchedulerJobRun（status=running）
2. 调用 seed_calendar_from_pytdx(session, year=当前年份)
3. 更新或插入交易日历记录
4. 完成后更新 SchedulerJobRun（status=succeeded/failed）
```

## 5. 盘中监控任务（monitor_scheduler）

### 5.1 调度配置

- **交易时段**：9:30-11:30 / 13:00-15:00
- **执行间隔**：30 秒
- **午休**：11:30-13:00 暂停
- **非交易日**：5 分钟检查一次

### 5.2 执行流程

```
1. 交易日判断（is_trading_day_async）
2. 交易时段判断（_get_monitor_session）
3. 查找或创建当前 session 的 SchedulerJobRun（同一 session 只创建一条）
4. MonitorBatchService.execute_monitor_cycle：
   a. 查询用户自选股去重
   b. WatchlistMonitor.calculate_state
   c. detect_events
   d. 写入 strategy_events（event_key 幂等）
   e. 写入 MonitorEvaluation（exactly-once）
   f. 周期结束后按用户合并为飞书卡片通知
   g. create_message + write_outbox
5. 更新 session 级统计（succeeded_count / failed_count / last_cycle_at）
6. session 接近结束时标记完成
```

### 5.3 session 聚合

- 同一业务日同一 session（morning/afternoon）只创建一条 `SchedulerJobRun`
- session 内持续更新 `last_cycle_at`、`succeeded_count`、`failed_count`
- session 结束前 `cycle_interval + 5` 秒标记为 `succeeded`

## 6. 策略批量任务（strategy_batch）

### 6.1 轮询配置

- **执行间隔**：5 秒（`WORKER_INTERVAL`）
- **单次批量**：1 条（`claim_next_run` 加锁领取）

### 6.2 执行流程

```
1. 启动时调用 recover_stale_runs 恢复过期租约
2. 循环：
   a. claim_next_run（SELECT FOR UPDATE SKIP LOCKED）
   b. 无任务时 sleep 5s
   c. execute_run：
      - 数据就绪检查
      - 逐标的执行策略（compute_dsa_history）
      - 写入 strategy_results
      - 质量门禁检查
      - 自动 publish
   d. 每 30s 更新心跳与租约
```

### 6.3 状态机

```
queued → running → completed/partial_failed → published/failed
                  → failed（终态）
```

### 6.4 租约与恢复

- **租约时长**：30 分钟（`_LEASE_DURATION_MINUTES`）
- **最大重试**：3 次（`_MAX_ATTEMPTS`）
- **stale 判定**：queued 状态超过 2 小时（`_STALE_QUEUED_HOURS`）
- **启动恢复**：`recover_stale_runs` 恢复过期 running 任务

## 7. Outbox Relay 任务（outbox）

### 7.1 轮询配置

- **执行间隔**：5 秒
- **单次批量**：100 条（`WORKER_BATCH_SIZE`）

### 7.2 执行流程

```
1. 查询 outbox WHERE status=pending
2. 普通事件：投递到 Redis 队列（LPUSH outbox:queue:{event_type}）
3. notification.message.created 事件：
   a. 查询通知的目标渠道
   b. 为每个渠道创建 MessageDelivery(pending)
   c. 标记 Outbox 为 processed
4. 失败 → retry_count + 1，保持 pending
```

## 8. 投递任务（delivery）

### 8.1 轮询配置

- **执行间隔**：5 秒
- **单次批量**：50 条（`DEFAULT_BATCH_SIZE`）
- **最大重试**：3 次（`DEFAULT_MAX_RETRY`）

### 8.2 重试退避

```python
_RETRY_BACKOFF_BASE = 30  # 第 n 次等待 2^n * 30 秒
```

### 8.3 静默时段

- **静默时段**：22:00-08:00（上海时区）
- **行为**：静默时段不投递，仅站内可见
- **下次投递**：`_compute_next_attempt_at` 计算静默结束后首个可投递时间

### 8.4 状态机

```
pending → sending → success（终态）
                  → failed → retrying → success / dead
                            → dead（超过最大重试）
```

## 9. 任务恢复机制

事实源：`backend/app/worker.py`

### 9.1 Worker 启动恢复

事实源：`backend/app/services/scheduler_job_run_recovery_service.py`

```python
# 所有 scheduler worker（bars/strategy/calendar/monitor）+ after_close_orchestrator 启动时调用
recover_stale_scheduler_job_runs(db)
# 原子更新：status=running 且 (lease_expires_at < now OR heartbeat_at < now - 90s) → interrupted
# heartbeat 阈值 90s = 3 个心跳周期（30s×3），不凭 created_at 强制中断
```

**4 个恢复调用点**（统一函数复用，避免逻辑漂移）：

1. `lifespan` 启动时（backend 进程）
2. `run_after_close_orchestrator_worker` 启动时
3. `acquire_job_run_lock` 抢锁前（幂等创建任务时先清理僵尸）
4. `run_watchdog` 每 60s 周期调用

### 9.2 盘后编排任务（after_close_orchestrator）

事实源：`backend/app/services/after_close_orchestrator.py` + `backend/app/worker.py`

- **WORKER_TYPE**：`after_close_orchestrator`
- **领取方式**：`FOR UPDATE SKIP LOCKED` 领取 `queued` 任务
- **心跳租约**：每 30s 更新 `heartbeat_at` 与 `lease_expires_at`（默认租约 `_ORCHESTRATOR_LEASE_SECONDS`）
- **断点恢复**：`metadata_json.last_completed_step` 记录已完成阶段，Worker 重启后跳过
- **3 个手动入口**（创建 queued 任务，由 Worker 领取执行）：
  - 更新日线 + DSA：`mode=refresh_and_dsa`
  - 仅重算 DSA：`mode=dsa_only`（跳过 `refresh_daily`，直接调用 `strategy_batch_service.create_batch_run`）
  - 从失败继续：`mode=resume_from_failure`
- **幂等键**：`run_key` 部分唯一索引（仅约束 `queued/running`），保证同一业务日不重复执行

### 9.3 监控评估恢复

```python
# monitor_scheduler 启动时调用
service.recover_stale_evaluations(db)
# 恢复过期租约的 PENDING 评估
```

### 9.4 策略 run 恢复

```python
# strategy_batch worker 启动时调用
service.recover_stale_runs(db)
# 恢复过期 running 和 stale queued 任务
```

## 11. 优雅退出

- **信号处理**：SIGTERM / SIGINT → 设置 `_shutdown` 标志
- **scheduler**：`scheduler.shutdown(wait=False)`
- **轮询 worker**：当前轮询完成后退出
- **心跳任务**：退出时标记 `status=stopped`
- **租约清理**：下次启动时由 `recover_stale_scheduler_job_runs` 处理
