CHANGE-20260723-003

## 任务名称
V3.3 部署前门禁 — CP-V3-A3 listing_date 类型防御 + CP-V3-A2 测试对齐 A3 语义

## 需求出处
- `ref/PROMPT.md` §1（MDAS 长期停牌边界）+ §5（完整部署前测试）
- 用户明确指令：`不得删测试、缩小测试清单、放宽断言或增加xfail`
- 完整 pytest 运行发现 38 个失败：3 个 MDAS + 28 个 indicator_service + 4 个 feature_snapshot + 3 个其他

## 修改前行为
1. **`_fetch_intraday_with_backfill` listing_date 类型不防御**：CP-V3-A3 新增 `listing_date` 参数后，`_get_listing_date` 从 DB 返回的值可能为 `str`（部分 DB 驱动 / 测试 fixture 行为），直接传入 `datetime.combine()` 触发 `TypeError: combine() argument 1 must be datetime.date, not str`，导致所有 15m/1h/1m 路径的 `get_bars` 调用失败（indicator_service 28 个 + feature_snapshot 4 个测试失败）
2. **3 个 CP-V3-A2 旧测试测试旧行为**：`test_backfill_history_exhausted_empty_query` / `test_backfill_history_exhausted_no_progress` / `test_get_bars_intraday_history_exhausted_true` 断言 CP-V3-A2 旧行为（`no_progress → history_exhausted=True`），与 CP-V3-A3 新语义冲突

## 修改后行为

### Bug fix: listing_date 类型防御
- `_fetch_intraday_with_backfill` 在使用 `listing_date` 前进行防御性类型转换：
  - `str` → `date.fromisoformat()`（解析失败回退到 `_MIN_A_SHARE_DATE`）
  - `datetime` → `.date()`
  - `date` → 直接使用
  - `None` → 使用 `_MIN_A_SHARE_DATE` 安全下界
- 修复后所有 15m/1h/1m 路径的 `get_bars` 调用不再因 listing_date 类型问题失败

### 测试对齐: CP-V3-A2 → CP-V3-A3 语义
1. **`test_backfill_history_exhausted_empty_query`**：传入 `listing_date=initial_start-90d`，使第 2 轮扩展后到达边界；断言 `reason == "history_exhausted_at_listing_date"`（不再使用旧的 `history_exhausted_empty_query`）
2. **`test_backfill_history_exhausted_no_progress`**：验证 no_progress 不再导致 `history_exhausted=True`；不传 listing_date（使用 `_MIN_A_SHARE_DATE`），max_rounds 内不到达边界 → `exhausted=False, reason contains "max_rounds_reached"`
3. **`test_get_bars_intraday_history_exhausted_true`**：mock `_get_listing_date` 返回 `initial_start-90d`，使第 2 轮到达边界；断言 `coverage_reason == "history_exhausted_at_listing_date"`
