# 安全规范

## 1. 密钥管理

### 1.1 密钥清单

事实源：`backend/.env.example` + `docker-compose.prod.yml`

| 密钥 | 用途 | 存储位置 | 轮换频率 |
|------|------|---------|---------|
| `JWT_SECRET` | JWT 签名 | env 文件 | 季度 |
| `SECRET_MASTER_KEY` | Secret 加密主密钥（Fernet 派生） | env 文件 | 季度 |
| `DATABASE_URL` 中的密码 | PostgreSQL 业务用户密码 | env 文件 | 半年 |
| 飞书 `app_secret` | 飞书平台应用凭证 | DB（加密存储） | 应用变更时 |
| 飞书 `sign_secret` | Webhook 签名密钥 | DB（加密存储） | 应用变更时 |

### 1.2 密钥存储

- **生产 env 文件**：`/etc/market-dev/market.env`，权限 `600`，仅 root 可读
- **禁止**：将真实密码写入代码、提交到 Git、记录到日志
- **禁止**：在 `.env.example` 中填写真实密码
- **DB 中的密钥**：通过 `encrypt_secret` 加密存储，读取时通过 `decrypt_secret` 解密

### 1.3 密钥派生

事实源：`backend/app/core/security.py`

**JWT 签名**：
```python
jwt.encode(payload, settings.jwt_secret, algorithm=settings.jwt_algorithm)
# 默认算法：HS256
```

**Fernet 加密**（用于 Secret 字段）：
```python
# 从 SECRET_MASTER_KEY 派生 Fernet 密钥
digest = hashlib.sha256(settings.secret_master_key.encode()).digest()
fernet_key = base64.urlsafe_b64encode(digest)
# Fernet 要求 32 字节密钥 urlsafe base64 编码
```

### 1.4 密钥轮换流程

**JWT_SECRET 轮换**：
1. 在 env 文件中新增 `JWT_SECRET_NEW=<new-secret>`
2. 修改 `app/core/deps.py` 支持 dual secret 验证（新旧都接受）
3. 重启 backend
4. 等待所有 access token 过期（最长 1 小时）
5. 移除旧 `JWT_SECRET`，将 `JWT_SECRET_NEW` 重命名为 `JWT_SECRET`
6. 重启 backend

**SECRET_MASTER_KEY 轮换**：
1. 编写迁移脚本：遍历所有加密字段，用旧密钥解密，用新密钥加密
2. 在 env 文件中设置新密钥
3. 执行迁移脚本
4. 重启 backend

> **重要**：SECRET_MASTER_KEY 轮换需要先迁移历史数据，否则无法解密。

| `bz` | `bz_stock` / `bz_stock_test` 全部权限 | 应用业务读写 |

### 2.2 角色创建


### 2.3 连接限制

- **生产库**：仅 127.0.0.1 监听，不暴露到公网
- **测试库**：容器内 5432 → 宿主机 5433，仅本地访问
- **应用连接**：通过 `DATABASE_URL` 环境变量，使用 `bz` 角色

### 2.4 最小权限原则

- 应用仅使用 `bz` 角色，不使用 `postgres` 超级管理员
- 测试库与生产库物理隔离（不同数据库 + 不同端口）
- 禁止应用账号执行 DDL（迁移由 Alembic 单独执行）

## 4. 认证与授权

### 4.1 JWT Token 类型

事实源：`backend/app/core/deps.py` + `security.py`

| Token 类型 | TTL | 用途 | 限制 |
|-----------|-----|------|------|
| `access` | 1 小时 | API 认证 | 用户状态必须 active |
| `refresh` | 7 天 | 刷新 access token | 仅用于 `/auth/refresh` |
| `capture` | 短期 | 截图模式 | 仅 `/stock/{symbol}?capture=feishu` |

### 4.2 RBAC 实现

事实源：`backend/app/core/deps.py`

```python
# 角色检查依赖工厂
require_roles("admin")  # 任一匹配即通过

# 用法
@router.post("/admin/invite-codes", dependencies=[Depends(require_roles("admin"))])
```

### 4.3 安全约束

- **用户 ID 注入**：私有资源的 `user_id` 由认证上下文注入，不接受客户端传入
- **状态检查**：`status != active` 的用户拒绝访问（403）
- **常量时间比较**：密码验证使用 `bcrypt.checkpw`（避免时序攻击）
- **密码长度限制**：bcrypt 限制 72 字节，超过截断

### 4.4 密码哈希

事实源：`backend/app/core/security.py`

```python
# 哈希
hashed = bcrypt.hashpw(plain.encode("utf-8")[:72], bcrypt.gensalt())

# 验证（常量时间比较）
bcrypt.checkpw(plain.encode("utf-8")[:72], hashed.encode("utf-8"))
```

**约束**：
- 直接使用 `bcrypt` 库（避免 passlib 1.7.4 与 bcrypt 5.x 兼容性问题）
- 自动处理盐与版本前缀（`$2b$`）
- 密码超过 72 字节时截断

## 5. Secret 加密

### 5.1 加密场景

- 飞书 `app_secret`：用户渠道配置中的敏感字段
- 飞书 `sign_secret`：Webhook 签名密钥

### 5.2 加解密 API

事实源：`backend/app/core/security.py`

```python
# 加密
cipher = encrypt_secret("my-secret-webhook-key")

# 解密
plain = decrypt_secret(cipher)
```

### 5.3 脱敏规则

事实源：`backend/app/schemas/notification.py`

API 读取渠道配置时：
- `app_secret` 仅显示末 4 位（如 `****abcd`）
- `sign_secret` 仅显示末 4 位
- 用户创建 / 更新时传入明文，写入时加密

## 6. 网络安全

### 6.1 端口暴露

| 端口 | 服务 | 暴露范围 | 说明 |
|------|------|---------|------|
| 80 | frontend (Nginx) | 公网 | SPA + `/api/` 反代 |
| 8000 | backend (FastAPI) | 本地 | 仅通过 nginx 反代访问 |
| 8001 | worker-capture | 本地 | 被 backend 调用 |
| 5432 | PostgreSQL | 本地 | 仅 127.0.0.1 |
| 5433 | 测试库 | 本地 | 仅 127.0.0.1 |
| 6379 | Redis | 容器网络 | 不暴露到宿主机 |

### 6.2 防火墙

- **UFW**：当前为 inactive，端口暴露由 Docker 管理
- **建议**：生产环境启用 UFW，仅开放 80（必要可加 443）和 SSH 端口

### 6.3 HTTPS（待实施）

当前测试环境为 HTTP，生产环境建议：
- 添加 HTTPS（Let's Encrypt 证书）
- 配置 nginx HSTS
- 禁用旧版 TLS（仅 TLS 1.2+）

## 7. 输入验证

### 7.1 Pydantic 校验

所有 API 请求体使用 Pydantic 模型校验：
- 类型检查
- 必填字段
- 字段约束（如 `min_length` / `max_length`）
- 自定义验证器

### 7.2 SQL 注入防护

- **强制使用 SQLAlchemy ORM**：禁止拼接 SQL 字符串
- **参数化查询**：使用 `select().where()` 等构造器
- **特殊场景**：必须使用 `text()` 时，参数通过 `params` 传入

### 7.3 XSS 防护

- **前端**：React 默认转义插值表达式
- **dangerouslySetInnerHTML**：禁止使用（除非经过严格清洗）
- **CSP**：建议在 nginx 配置 Content-Security-Policy

## 10. 安全事件响应

### 10.1 密钥泄露

1. 立即轮换泄露的密钥（参考第 1.4 节）
2. 检查日志中是否有异常访问
3. 通知所有用户重新登录（轮换 JWT_SECRET 后强制下线）
4. 复盘泄露原因，修复流程

### 10.2 数据库被入侵

1. 立即修改 `bz` 用户密码
2. 检查 `scheduler_job_runs` / `worker_heartbeats` 是否有异常记录
3. 检查 `users` 表是否有未授权用户
4. 检查 `notification_channels` 是否有可疑渠道
5. 从备份恢复（如必要）

### 10.3 用户账号被入侵

1. 禁用账号：`UPDATE users SET status = 'disabled' WHERE email = 'xxx';`
2. 撤销该用户的所有 token（轮换 JWT_SECRET，影响所有用户）
3. 通知用户重置密码
4. 检查该用户的操作记录
