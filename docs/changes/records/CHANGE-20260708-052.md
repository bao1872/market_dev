# CHANGE-20260708-052

## 变更编号
CHANGE-20260708-052

## 任务名称
盘后流水线可视化面板

## 需求出处
管理员需要在 UI 上查看收盘后增量更新是否执行、执行到哪一步、是否可供自选股读取。

## 修改前行为
系统概览页面有 `AfterClosePipelineCard` 组件，但缺少完整的盘后流水线详情页面。管理员无法直观看到 8 步骤时间线、数据新鲜度、事件日志、watchlist_ready 判定等聚合状态。

## 修改后行为
1. 新增 admin-only 聚合 API（复用现有 service，不新建平行接口）：
   - `GET /admin/after-close/pipeline/latest`
   - `GET /admin/after-close/pipeline?trade_date=YYYY-MM-DD`
   - `GET /admin/after-close/pipeline/runs?limit=20`
   - `POST /admin/after-close/pipeline/run`（幂等：同 trade_date 已有 queued/running/succeeded 时返回 existing）

2. API 返回聚合状态：trade_date、market_session、overall_status、watchlist_ready、8 步骤状态、数据新鲜度、事件日志（最多 100 条）

3. watchlist_ready 严格复用现有 gate 语义：`status='succeeded' AND published_at IS NOT NULL AND metadata_.scope='full'`，sample backfill 不显示为前台可读

4. 状态文案区分：未开始/运行中/成功/手动补齐/sample 验证/失败/阻塞

5. `get_latest_pipeline` 策略：交易日（含今日）始终以 today 为目标 trade_date，即使无 after_close run 也返回 today 的 not_started/blocked，不回退历史 run 掩盖"今天未执行"；非交易日回退到最近有记录的交易日

6. blocked 判断使用 `MARKET_SESSION_CLOSED` 常量（从 `market_status_service` 导入），收盘后超过 30 分钟阈值仍无 run → blocked

7. `feature_snapshot_run` 摘要优先返回 succeeded+published+full run（即 watchlist_ready 的实际数据源），若不存在再 fallback 到最新任意 run；避免 watchlist_ready=true 但页面展示 sample run 的误导

8. `PipelineStep.counts` 使用 `Field(default_factory=dict)`，避免可变默认值共享问题

9. 前端新增 `/admin/after-close` 详情页：顶部状态卡、8 步骤时间线、数据新鲜度卡、最近 20 次运行列表、事件日志抽屉

10. 系统概览 `AfterClosePipelineCard` 改为摘要卡，提供进入 `/admin/after-close` 链接

11. 轮询：running 状态 10 秒一次，非 running 60 秒一次，页面不可见暂停轮询
