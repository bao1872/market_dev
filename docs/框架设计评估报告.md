## 二、六维对比矩阵

### 2.1 架构模式

| 维度 | V1.1 实现 | V1.0 参考 | 评分 |
|---|---|---|---|
| **分层粒度** | 四层分离（api/services/repositories/models），提交仅在 API 层，服务层只 flush 获取 ID | 两层（api/routes.py 直接查 ORM），无独立 repository 层 | V1.1 优 |
| **策略插件** | StrategyRuntime ABC（selector + monitor 双范式）+ StrategyLoader 动态注册 + BudgetGuard 资源预算 + 事件注册表 | 无策略插件抽象，`make_chart` 和 `calculate_mock_signal` 为硬编码适配器 | V1.1 优 |
| **分布式原语** | Redis 队列（LPUSH/BRPOP）+ Lua 脚本分布式锁 + 事务性 Outbox + 幂等键 SET NX EX | 无队列/锁/Outbox，worker 为单进程 `while True` + `time.sleep` | V1.1 优 |
| **事务边界** | API 层 commit，服务层 flush，异常链式 re-raise | 路由内 commit，`with_for_update()` 行锁用于邀请码兑换 | V1.1 优 |

**事实依据**：
- V1.1：[strategies.py](file:///root/web_dev/backend/app/api/strategies.py) API 层依赖 schemas + services；[strategy_service.py](file:///root/web_dev/backend/app/services/strategy_service.py) 服务层拥有状态机 + 类型化异常
- V1.0：[routes.py](file:///root/web_dev/ref/fullstack_ref/fullstack_app/backend/app/api/routes.py) 第 189-212 行 screener 查询内联在路由中；[worker.py](file:///root/web_dev/ref/fullstack_ref/fullstack_app/backend/app/worker.py) 第 43-46 行 `while True` 循环

### 2.3 数据流

| 维度 | V1.1 实现 | V1.0 参考 | 评分 |
|---|---|---|---|
| **端到端流** | DB → repositories（async + pytdx 回填）→ services（状态机）→ api（Pydantic 序列化）→ HTTP → axios → Zustand/React Query → React | DB → ORM → routes.py（内联查询）→ HTTP → fetch → useState → React | V1.1 优 |
| **策略执行流** | StrategyVersion → StrategyLoader.load → runtime.initialize → BudgetGuard.run_with_budget → StrategyResult → strategy_results 表 → selection_executor → selection_composer → SelectionPlanRun | 无策略执行流，`make_chart` 硬编码生成 | V1.1 优 |

| **异步处理** | 全链路 async（asyncpg + async SQLAlchemy + asyncio.to_thread 桥接同步 pytdx） | 同步 SQLAlchemy，worker 为同步循环 | V1.1 优 |

## 三、关键决策点

### 决策点 1：分层粒度

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | 四层分离（api/services/repositories/models），提交仅在 API 层 |
| **选项 B（V1.0）** | 两层（api/routes.py 直接查 ORM），单文件 559 行 |
| **权衡** | A 牺牲导航便利性换取可维护性和可测试性；B 牺牲可维护性换取单文件可读性 |
| **结论** | **选 A**。项目需求含多用户共享计算 + 策略插件化 + 监控状态机，业务复杂度高，四层分离是必要的。V1.0 的单文件路由在 559 行已接近可维护上限，无法承载 V1.1 的 22 份文档规格。 |

### 决策点 2：状态管理

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | Zustand 4.5（3 个独立 store）+ @tanstack/react-query 5.51（服务端状态） |
| **选项 B（V1.0）** | React Context（AuthContext + Toast）+ useState/useEffect（页面本地状态） |
| **权衡** | A 提供独立 store 避免不必要的重渲染 + 服务端状态缓存；B 更简单但页面间数据不共享 |
| **结论** | **选 A**。选股结果、监控状态、消息列表需跨页面共享，Zustand 独立 store + React Query 缓存是必要的。V1.0 的 Context 方式在页面增多后会导致重复请求。 |

### 决策点 3：任务队列

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | Redis 队列（LPUSH/BRPOP）+ Lua 脚本分布式锁 + 事务性 Outbox + 幂等键 |
| **选项 B（V1.0）** | 单进程 `while True` + `time.sleep`，无队列/锁 |
| **权衡** | A 支持多 worker 水平扩展 + 至少一次投递 + 幂等；B 简单但无法扩展 |
| **结论** | **选 A**。监控需 1 分钟 Bar 驱动全市场扫描，单进程无法满足 SLA。Redis 队列 + 分布式锁支持多 worker 并行消费，Outbox 保证消息不丢。 |

### 决策点 4：迁移策略

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | 14 个有序 Alembic migration，含 upgrade + downgrade，手写（target_metadata=None） |
| **选项 B（V1.0）** | `Base.metadata.create_all()` 启动建表，无迁移 |
| **权衡** | A 支持版本化 schema 演进 + 回滚；B 简单但无法演进 |
| **结论** | **选 A**。生产系统必须支持 schema 演进。V1.1 的手写 migration 虽无 autogenerate 但有完整 upgrade/downgrade，可接受。建议后续启用 autogenerate 做 ORM-DB 漂移检测。 |

### 决策点 5：部署方式

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | 无生产部署工件，仅开发 docker-compose（db + redis） |
| **选项 B（V1.0）** | 完整 Docker Compose 4 服务 + Nginx + 多阶段 Dockerfile |
| **权衡** | A 缺失部署能力；B 提供一键启动但架构简单 |
| **结论** | **选 A 架构 + 借鉴 B 部署模式**。V1.1 架构正确但部署缺失，需从 V1.0 借鉴 Dockerfile + nginx + 生产 compose，适配 V1.1 的多 Worker 架构（Selector/Monitor/Outbox Relay/Delivery Worker 独立进程）。 |

### 决策点 6：前端图表

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | lightweight-charts 4.2（TradingView 风格），已声明但未实现 |
| **选项 B（V1.0）** | Canvas 手绘 274 行，含 K 线/成交量/VWAP/ATR Rope/Volume Profile/POC/VAH/VAL/Delta/事件标记 |
| **权衡** | A 专业图表库，交互丰富但需适配；B 自定义灵活但维护成本高 |
| **结论** | **选 A**。lightweight-charts 提供专业 K 线交互（缩放/十字线/OHLC 联动），V1.0 的 Canvas 实现可作为图层绘制参考。策略级图层控制（V1.6.1）需基于图表库 API 实现。 |

### 决策点 7：安全实现

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | JWT（access + refresh）+ bcrypt + RBAC 中间件 + UserContext 注入 + Fernet 加密 + 脱敏返回 |
| **选项 B（V1.0）** | JWT（无 refresh）+ scrypt + `hmac.compare_digest` 时序安全比较 + Fernet 加密 + 脱敏返回 |
| **权衡** | A 有 refresh token + RBAC 中间件；B 有 scrypt（更安全）+ 时序安全比较 |
| **结论** | **选 A 为主 + 借鉴 B 的 scrypt**。V1.1 的 refresh token + RBAC 中间件是生产必需。V1.0 的 scrypt 比 bcrypt 更抗 GPU 破解，可在后续迭代中替换。 |

### 决策点 8：测试策略

| 项 | 内容 |
|---|---|
| **选项 A（V1.1）** | 311 个测试函数 + 100% 模块自测 + 合成数据（固定 seed）+ 边界覆盖 |
| **选项 B（V1.0）** | 3 个测试函数 + 无模块自测 |
| **权衡** | A 全面但开发成本高；B 快速但覆盖不足 |
| **结论** | **选 A**。策略插件 + 组合引擎 + 状态机 + 通知链路复杂度高，必须有全面测试。V1.1 的 311 测试 + 100% 自测是正确的工程实践。 |

## 四、选型结论

### 4.1 选定方向

**选定 V1.1 企业级分层架构作为主框架，借鉴 V1.0 的部署完备性模式。**

### 4.2 技术依据

1. **业务复杂度匹配**：V1.1 文档集含 22 份 md + 6 份 schema + DDL + OpenAPI，涵盖 R1-R9 基础设施、S1-S2 选股、M1-M4 监控、C1-C9 组合与通知。V1.0 的单文件路由（559 行）无法承载此规格，V1.1 的四层分离是必要的。

2. **可扩展性需求**：监控需 1 分钟 Bar 驱动全市场扫描（数千只股票），单进程无法满足 SLA。V1.1 的 Redis 队列 + 分布式锁 + Outbox 支持多 worker 水平扩展。

3. **策略插件化**：V1.1 的 StrategyRuntime ABC（selector + monitor 双范式）+ StrategyLoader 动态注册 + BudgetGuard 资源预算是策略插件化的正确抽象。V1.0 无此能力。

4. **可维护性**：V1.1 的 311 测试 + 100% 模块自测 + Alembic 迁移 + 精确依赖锁定是生产级可维护性保障。V1.0 的 3 测试 + create_all + latest 依赖无法满足。

5. **异步性能**：V1.1 全链路 async（asyncpg + async SQLAlchemy + asyncio.to_thread）避免事件循环阻塞。V1.0 同步处理在高并发下会成为瓶颈。

### 4.3 项目需求匹配度

| 项目需求 | V1.1 匹配度 | V1.0 匹配度 |
|---|---|---|
| 多用户共享计算 | 高（Redis 队列 + 分布式锁） | 低（单进程） |
| 策略插件化 | 高（StrategyRuntime ABC） | 无 |
| 监控实时性 | 高（1m Bar + 状态机 + Outbox） | 低（60s 轮询） |
| 飞书通知 | 高（Channel Adapter + 幂等 + 重试） | 中（有飞书但无重试/限流） |
| 数据新鲜度 SLA | 高（1800s/90s 检查） | 无 |
| 生产部署 | 低（无 Dockerfile） | 高（完整 Docker Compose） |

### 4.4 借鉴 V1.0 的改进点

V1.0 在部署完备性上领先，以下改进点需融入 V1.1：

1. **后端 Dockerfile**：多阶段构建（builder + runtime），清华源
2. **前端 Dockerfile**：多阶段构建（builder + nginx runtime）
3. **生产 docker-compose.yml**：5 服务（backend + worker + frontend + db + redis）
4. **Nginx 反向代理**：`/api` → backend:8000，静态资源 → frontend
5. **schema-driven 前端表格**：V1.0 的 `ScreenerPage` 从 `result_schema` 动态生成列，V1.1 可借鉴
6. **Worker 独立进程**：V1.0 的 docker-compose worker 服务模式，适配 V1.1 的多 Worker
