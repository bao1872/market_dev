# CHANGE-20260707-047: Feature Snapshot 持久化（自选股监控指标从实时计算切换为盘后快照）

## 变更编号
CHANGE-20260707-047

## 任务名称
Feature Snapshot 持久化：将自选股 `/watchlist/monitor-status` 与历史因子探索从"按需实时计算"切换为"盘后快照持久化 + 历史回补"。

## 需求出处
- 盘迹 PR #34 后的下一阶段：盘后特征快照持久化（计划文档：`.trae/documents/feature_snapshot_persistence_plan.md`）。
- 自选股监控页面在交易时段多次刷新会触发重复全市场结构因子计算，性能浪费且数据口径不稳定。
- 需要保留历史交易日的特征因子，支持后续历史回溯与归因。

## 修改前行为
- `GET /watchlist/monitor-status` 的 metrics 来自 `MonitorSnapshotService` 实时计算 + `MonitorState.payload` fallback：
  - `backend/tests/test_watchlist_monitor_status_fallback.py` 描述旧契约。
- 盘后编排状态机：`queued → refreshing_daily → checking_coverage → creating_dsa → waiting_dsa_worker → quality_gate → publishing → succeeded`，无 `feature_snapshot` 步骤。
- 历史交易日无任何特征快照落盘，无法回溯。
- `MonitorEvaluation` 既是评估状态来源，也作为 metrics fallback 数据源。

## 修改后行为

### 1. 新增 `stock_feature_snapshots` 表
- 主键 `id`（UUID，默认 `gen_random_uuid()`）。
- 业务唯一键 `(instrument_id, trade_date, primary_timeframe, secondary_timeframe, adj, schema_version)`（UNIQUE 约束 `uq_feature_snapshot_instrument_date_tf_adj_schema`）。
- 时间戳字段：
  - `source_primary_bar_time` `timestamptz`：日线规范化为 `trade_date 15:00+08:00`。
  - `source_secondary_bar_time` `timestamptz`：15m 取最后一根 15m bar 的实际 trade_time。
- payload 字段：
  - `structural_payload` JSONB：`_compute_all_factors_for_bars` 完整输出（primary + secondary + meta）。
  - `temporal_payload` JSONB：`_compute_daily_context` / `_compute_m15_response` / `_compute_derived_relation` 完整输出。
  - `summary_payload` JSONB：前端列表用摘要，含 `_source='feature_snapshot'`、`poc_price`、`nearest_node_above/below`、`distance_to_node_*_atr`、`node_interval_position_0_1`、`cost_position_zone`、`value_area_zone`、`daily/m15_developing_swing_dir/high/low`、`m15_position_relative_to_daily`、`current_price`、`change_pct`、`bb_upper/mid/lower`、`as_of`、`source_bar_time`。
  - `degraded_reasons` JSONB 默认 `'[]'`：单股失败不阻断其他股票。
- 索引：
  - `ix_feature_snapshot_trade_date_schema (trade_date, schema_version)`：按日查询快照。
  - `ix_feature_snapshot_instrument_date (instrument_id, trade_date desc)`：单股历史回溯。
  - `ix_feature_snapshot_date_instrument (trade_date, instrument_id)`：按日批量扫描。
- 不给 full payload 加 GIN 索引（节省磁盘）。

### 2. 新增 `feature_snapshot_service.py`
- 常量：`_SHANGHAI_TZ = ZoneInfo("Asia/Shanghai")`、`_SCHEMA_VERSION = 1`、`_PRIMARY_LOOKBACK = 500`、`_SECONDARY_LOOKBACK = 500`、`_BB_WIN = 20`、`_BB_K = 2.0`。
- 纯函数：
  - `_truncate_bars_to_trade_date(bars, trade_date, timeframe)`：按 `index.date <= trade_date` 截断，保证 point-in-time。
  - `build_summary_payload(structural, temporal, trade_date, source_bar_time, extra)`：抽取前端列表用字段，缺字段填 `None`，附 `_source='feature_snapshot'`。
  - `_extract_extra_fields(df_1d)`：从日线最后一根 bar 提取 `current_price` / `change_pct` / `bb_upper/mid/lower`（BB 用 `bollinger_features_plotly.bollinger(bars, 20, 2.0)`）。
  - `_normalize_primary_bar_time(df_1d, trade_date)`：1d bar 时间规范化为 `trade_date 15:00+08:00` aware datetime。
  - `_normalize_secondary_bar_time(df_15m)`：15m bar 时间取最后一根 15m bar 实际 trade_time，转 Asia/Shanghai aware。
- 异步函数：
  - `compute_feature_snapshot_for_date(session, instrument_id, trade_date, primary_timeframe='1d', secondary_timeframe='15m', adj='qfq', primary_bars=None, secondary_bars=None)`：单股单日 point-in-time 计算，复用 `_compute_all_factors_for_bars`、`_compute_daily_context`、`_compute_m15_response`、`_compute_derived_relation`、`bollinger()`，不复制公式。
  - `upsert_snapshot(session, snapshot)`：按唯一键 `ON CONFLICT DO UPDATE`，`structural_payload/temporal_payload/summary_payload/source_*_bar_time/degraded_reasons` 覆盖，`updated_at = now()`。
  - `get_active_a_share_instruments(session)`：查询所有 active A 股 instrument_id 列表（不含 B 股/指数/退市）。
  - `compute_for_trade_date(session, trade_date, instrument_ids, batch_size=20, commit_every=500, failure_threshold=0.3)`：批量计算单日所有 instrument 的 snapshot，单股失败写 `degraded_reasons` 不阻断其他股票，失败比例超过 30% 抛 `RuntimeError`。

### 3. 修改 `after_close_orchestrator.py` 状态机
- 新状态流转：`queued → refreshing_daily → checking_coverage → creating_dsa → waiting_dsa_worker → quality_gate → feature_snapshot → publishing → succeeded`。
- `AfterCloseRunStatus` 新增 `FEATURE_SNAPSHOT = "feature_snapshot"`。
- 在 quality_gate 与 publishing 之间新增步骤 3.5：
  - 使用独立 `AsyncSessionLocal` 调用 `get_active_a_share_instruments` + `compute_for_trade_date`。
  - 单股失败不阻塞，失败比例超过 30% 抛 `RuntimeError` 标记 `failed`。
  - 完成后更新心跳与 `last_completed_step='feature_snapshot'`。
- 断点恢复：`_completed_steps` 字典新增 `feature_snapshot` 节点，后续 `publishing` / `succeeded` 节点依赖集合中加入 `feature_snapshot`。

### 4. 修改 `/watchlist/monitor-status` API
- metrics 数据源唯一为 `StockFeatureSnapshot.summary_payload`，删除 `MonitorSnapshotService` 实时 fallback 与 `MonitorState.payload` fallback。
- `MonitorEvaluation` 仅用于展示评估状态字段（`evaluation_status` / `retry_count` / `error_code` / `source_bar_time`），不再作为 metrics 来源。
- 新增 `_get_expected_snapshot_trade_date(today, market_session, is_trading_day)`：
  - 非交易日 → `None`（前端展示 `NO_SNAPSHOT`）。
  - 交易日但未收盘（PRE_OPEN/MORNING_SESSION/LUNCH_BREAK/AFTERNOON_SESSION）→ `None`（前端展示 `NO_SNAPSHOT`）。
  - 交易日已收盘（MARKET_CLOSED）→ `today`（snapshot 缺失时前端展示 `WAITING_SNAPSHOT`）。
- 响应字段 `calculation_status` 三值：
  - `SUCCEEDED`：snapshot 存在，metrics 来自 `summary_payload`，`freshness_seconds` 基于 `snapshot.updated_at`。
  - `WAITING_SNAPSHOT`：交易日已收盘但 snapshot 缺失（orchestrator 未跑或失败）。
  - `NO_SNAPSHOT`：非交易日或交易日内（snapshot 不应存在）。
- `monitor_status` 兼容字段：SUCCEEDED 时回落到 `market_session`，否则与 `calculation_status` 一致。
- `freshness_seconds` 基于 `snapshot.updated_at` 与当前上海时间差（秒）。

### 5. 新增 `scripts/feature_snapshot_backfill.py` 历史回补脚本
- CLI 参数：
  - `--start` (必填)：起始日期 YYYY-MM-DD。
  - `--end` (默认 `latest`)：结束日期或 `latest`（解析为 `bars_daily` 表最新 trade_date）。
  - `--batch-size` (默认 20)：每批 instrument 数。
  - `--commit-every` (默认 500)：每 N rows commit 一次。
  - `--resume`：跳过已有 snapshot 的 (instrument_id, trade_date) 组合。
  - `--dry-run`：只打印计划，不执行写入。
  - `--failure-threshold` (默认 0.3)：失败比例阈值。
- 核心函数：
  - `get_trade_dates_from_bars(db, start, end)`：从 `bars_daily` 表查询已有交易日（确保数据存在再计算）。
  - `get_latest_bar_date(db)`：查询 `bars_daily` 表最新交易日。
  - `count_existing_snapshots(db, trade_date)`：统计某日已有 snapshot 数（用于 `--resume` 跳过判断）。
  - `backfill_single_date(db, trade_date, batch_size, commit_every, resume, dry_run, failure_threshold)`：调用 `compute_for_trade_date` 处理单日。
  - `main(args)`：遍历 trade_dates，逐日调用 `backfill_single_date`，单日失败不阻断其他日期。
- `start > end` 直接 `sys.exit(1)`。

## 风险
1. **历史数据缺口**：迁移上线后历史交易日无 snapshot，需运行 `feature_snapshot_backfill.py` 回补；回补前 `/watchlist/monitor-status` 在交易日收盘后会显示 `WAITING_SNAPSHOT`。
2. **orchestrator 失败传播**：`feature_snapshot` 步骤失败比例超过 30% 会让整个盘后流程标记 `failed`，可能导致 publishing 步骤不执行；需配合 monitor 告警。
3. **磁盘占用**：`structural_payload` / `temporal_payload` JSONB 体积较大，每只股票每个交易日约 10-20 KB；5000 股 × 250 天 ≈ 25-50 GB，需观察 `pg_total_relation_size('stock_feature_snapshots')` 增长。
4. **summary_payload 字段稳定性**：前端依赖字段名（`poc_price`、`daily_developing_swing_dir` 等），未来 schema 变更需 bump `schema_version` 并保证向后兼容。
5. **snapshot 与实时计算口径差异**：snapshot 使用 `include_realtime=False` 只取已完成 bar，而原实时计算可能含 partial bar；交易时段 `NO_SNAPSHOT` 期间用户看不到当日指标，是预期行为。

### 历史表述更正
- **错误表述**：PR body 与本文档早期版本曾称"原来 monitor-status 实时调用 `feature_snapshot_service`"。
- **真实历史**：原 `/watchlist/monitor-status` 的 metrics 来自 `MonitorSnapshotService` 实时计算 + `MonitorState.payload` fallback（与 `feature_snapshot_service` 无关，`feature_snapshot_service` 是本 PR 新增服务）。`feature_snapshot_service` 不存在"原来实时调用"一说。
- **修改前行为章节**的描述是正确的（`MonitorSnapshotService` 实时计算 + `MonitorState.payload` fallback），PR body 表述错误已纠正。

### Blocker 1: `_resolve_expected_snapshot_trade_date` 规则修正
- **问题**：原 `_get_expected_snapshot_trade_date` 只在 `MARKET_CLOSED` 返回 `today`，其余返回 `None`，导致盘中和非交易日永远 `NO_SNAPSHOT`。
- **修复**：重命名为 `_resolve_expected_snapshot_trade_date`（async），复用 `calendar_service`，规则如下：
  - 交易日未收盘 → 上一交易日（`get_previous_trading_day_async`）
  - 交易日已收盘（`MARKET_CLOSED`）→ `today`
  - 非交易日 → 最近交易日（`get_most_recent_trading_day_async`）
  - 无法解析最近交易日 → `None`
- **禁止硬编码周末**，必须查 `trading_calendar` 表。

### Blocker 2: 半成品 snapshot 事务边界修复
- **问题**：原 `compute_for_trade_date` 内部 `commit_every` 会提前 commit，最后才判断 `failure_threshold`，半成品落库后 watchlist 会读取。
- **修复**：
  1. `compute_for_trade_date` **不内部 commit**，只 upsert + flush + 阈值检查；`commit_every` 参数已删除。
  2. caller 决定 commit/rollback：
     - `after_close_orchestrator`：成功 commit；`RuntimeError` 时显式 `await db.rollback()` 回滚半成品，不进入 publishing。
     - `feature_snapshot_backfill`：每个 trade_date 一个独立事务，成功 commit，失败 rollback，单日失败不阻断其他日期。
  3. watchlist 只能读取成功完成的日期（V1 最低要求：失败日期不应有部分已 commit 行）。
- **新增测试**：`test_compute_for_trade_date_over_threshold_no_partial_after_rollback`、`test_execute_feature_snapshot_failure_skips_publishing`、`test_main_single_date_failure_rolls_back_and_continues`。

### Blocker 3: backfill resume 真正跳过 + dry-run 统计
- **问题**：原 `--resume` 仅调用 `count_existing_snapshots` 用于日志，仍 upsert 覆盖全部 instrument；dry-run 输出不足。
- **修复**：
  1. 新增 `get_existing_instrument_ids(db, trade_date, *, tf, adj, schema_version)` 返回已存在 instrument_id 集合（按完整唯一键过滤）。
  2. `backfill_single_date` 在 `--resume` 模式下从 active 列表过滤已存在 instrument，只对 missing 列表调用 `compute_for_trade_date`；全部已存在时跳过 compute。
  3. dry-run 输出 per-trade_date：active 数量、missing 数量、skipped_existing；summary 输出：trade_dates 数量、max active instruments、total_missing（estimated rows）。
  4. **Known Gap**：instrument-first 优化（按 instrument 批量取 bars，内存按 trade_date slice）本 PR 未实现，拆下一 PR；docs 标为 Known Gap，禁止全量生产 backfill。
