# CHANGE-20260709-006: after_close feature_snapshot 心跳保活 + stuck running snapshot run 修复

## 变更编号

CHANGE-20260709-006

## 任务名称

after_close_orchestrator feature_snapshot 阶段心跳保活与中断后 snapshot run 收口修复

## 需求出处

生产现场：2026-07-09 `after_close_orchestrator` 在 `feature_snapshot` 阶段被 watchdog/recovery 标记为 `interrupted`，但同 trade_date 的 `stock_feature_snapshot_run` 仍卡在 `running`，导致 UI 第 6 步显示异常且可能阻塞后续 after_close 重试。

## 修改前行为

- `feature_snapshot` 阶段没有独立后台心跳，长计算期间 `heartbeat_at` 不更新，租约过期后被 recovery worker 标记为 `interrupted`；
- `compute_for_trade_date` 无法向 orchestrator 汇报进度；
- orchestrator 中断后，`stock_feature_snapshot_run` 没有收口机制，可能长期 `running`；
- 新的 after_close 启动前不会清理 stuck running snapshot run，可能触发 partial unique index 冲突；
- `/admin/after-close/pipeline` 对 `interrupted + snapshot_run running` 的场景，第 6 步显示 `pending`，且未暴露 `feature_snapshot_run_id`/`feature_snapshot_progress`。

## 修改后行为

- `feature_snapshot` 阶段启动 `_job_run_heartbeat_loop`（间隔 30s），长计算期间持续刷新 `heartbeat_at`/`lease_expires_at`；
- `feature_snapshot_service.compute_for_trade_date` 支持可选 `progress_callback`，每处理完一个 batch 调用，回调更新 `metadata.feature_snapshot_progress` 并按每 500 只股票采样写入 `job_run_events`；
- 新增 `repair_stale_after_close_snapshot_runs`：仅当存在 `interrupted`/`failed` 的 after_close job_run 且同 trade_date 的 after_close snapshot_run 已超时，按实际 snapshot 行数与 `expected_count` 比例决定标记为 `succeeded`（≥95%）或 `failed`；
- `execute_after_close_run` 启动前自动调用 repair，避免 stuck running run 阻塞新任务；
- `/admin/after-close/pipeline` 在 `interrupted + snapshot_run running` 时，第 6 步显示 `running`，返回 `feature_snapshot_lost_contact=true`，并在 `after_close_run` 摘要中暴露 `feature_snapshot_run_id` 与 `feature_snapshot_progress`。

## 风险

- repair 按 95% 阈值将部分完成的 snapshot run 标记为 succeeded，存在少量缺失股票仍被 watchlist 读取的风险；缺失股票由单股 `degraded_reasons` 兜底，整体可用；
- 若 `feature_snapshot` 计算实际仍在进行但 orchestrator 已被 interrupted，repair 会强制 finish 该 run，可能丢失正在计算的进度；修复后通过新 heartbeat 机制避免；
- 生产修复前必须确认 `trading-worker-after-close` 已停止或已部署修复版本，否则 repair 可能与仍在写入的进程冲突。

## 遗留问题

- 当前生产 2026-07-09 stuck snapshot run 的实际 snapshot 行数未知，repair 结果可能是 `succeeded` 或 `failed`；若为 `failed`，需重跑 feature_snapshot；
- 若 feature_snapshot 阶段本身耗时过长（>4h），除 heartbeat 外仍需后续性能诊断；
- UI 文案“快照计算失联/待修复”需前端在 `/admin/after-close` 页面根据 `feature_snapshot_lost_contact` 渲染（后端已返回字段）。
