# CHANGE-20260707-040: DSA Overlay Source Alignment

## 变更编号
CHANGE-20260707-040

## 任务名称
DSA overlay 数据源对齐修复（15m/1h 误报"DSA 数据源不一致" + 03:00 时区错误）

## 需求出处
用户在 PR #29（developing swing）后测试发现非日线周期（特别 15m）出现 "DSA 数据源不一致，已暂停渲染"。15m 图顶部还显示 `2026-07-07 03:00`，对 A 股不合理。怀疑 chart bars 与 DSA overlay 的 timestamp/timezone/source/adjust 不一致。

## 修改前行为
- 后端 `_df_to_responses` 对 15m/1h 的 `trade_time` 返回 naive datetime（`ts.to_pydatetime()`），序列化为 `2026-07-06T15:00:00`（无时区后缀）。
- 前端 `new Date("2026-07-06T15:00:00")` 在 `America/New_York` 浏览器中把 naive ISO 当作本地时间解析 → `2026-07-06T15:00:00-04:00` (UTC=19:00) → 转 Asia/Shanghai = `2026-07-07 03:00+08:00` → 显示 `2026-07-07 03:00` 错误时间。
- 后端 `compute_source_bar_times(daily_bars)` 和 `compute_source_bar_hash(daily_bars)` 永远使用 `daily_bars` 计算 source_bar_times/source_bar_hash，且永远输出 `YYYY-MM-DD` 日期格式。
- 15m/1h 图表 `source_bar_times` 是日线日期格式（如 `"2025-06-25"`），与 K线 `trade_time`（`"2026-07-06T15:00:00"` 或 `+08:00`）格式不匹配。
- 前端 `normalizeChartTime` 对 15m 要求 `HH:MM`，日线日期返回 `null` → `indicatorKeys` 为空 → `matched=0` → `ratio=0` → 触发 "DSA 数据源不一致" banner。

## 修改后行为
- 后端 `_df_to_responses` 对 15m/1h 的 `trade_time` 返回 aware datetime（`ts.tz_localize("Asia/Shanghai")`），序列化为 `2026-07-06T15:00:00+08:00`，前端 `new Date(...)` 正确解析为 UTC 时刻，再转 Asia/Shanghai 显示 `2026-07-06 15:00` 正确北京时间。
- 1d `trade_date` 仍为 date 对象（无时区），向后兼容。
- 后端 `compute_source_bar_times(df, timeframe)` 和 `compute_source_bar_hash(df, timeframe)` 新增 `timeframe` 参数：
  - 1d/1w/1mo 用 `%Y-%m-%d`（YYYY-MM-DD，向后兼容）
  - 15m/1h 用 `%Y-%m-%dT%H:%M:%S`（YYYY-MM-DDTHH:MM:SS，无时区后缀）
- 后端 `indicator_service.compute_all_indicators` 在 15m/1h 改用 `macd_bars`（当前 timeframe bars）计算 source_bar_times/source_bar_hash，与 chart bars 同源（之前永远用 `daily_bars`）。
- 前端 `normalizeChartTime`（迁移到 `frontend/src/utils/chartTime.ts` 纯 .ts 模块便于 Node 测试）通过正则提取 date+HH:MM 前缀，忽略 `+08:00` 时区后缀和秒数，使 K线（aware）与 `source_bar_times`（naive）产生相同 canonical key。
- 15m 图不再误报 "DSA 数据源不一致"（matched/klineKeys.size = 1.0）。
- 故意构造的 source mismatch（如 15m source 仍是日线日期格式）仍触发 banner，保护 DSA 数学正确性。

## 风险
- 15m/1h `trade_time` 序列化格式从 naive `2026-07-06T15:00:00` 变为 aware `2026-07-06T15:00:00+08:00`，前端旧代码若用字符串前缀匹配可能受影响（已扫描前端 chart.ts `b.trade_time || b.trade_date` 不受影响）。
- `source_bar_times` 格式从永远 `YYYY-MM-DD` 变为按 timeframe 格式化（15m/1h 含时间），旧前端若假设 source_bar_times 永远是日期会受影响（已扫描前端 `normalizeChartTime` 处理 15m 含时间场景，向后兼容）。
- `indicator_service` 在 15m/1h 改用 `macd_bars` 计算 source 字段，影响 `source_bar_hash` 值（之前是日线 hash，现在是 15m/1h hash），前端 mismatch 校验变得更严格、更准确（这正是修复目标）。
- 性能：无新增表/worker/缓存；只单只股票 API 字段格式调整。
