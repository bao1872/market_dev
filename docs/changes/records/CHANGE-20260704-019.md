# CHANGE-20260704-019

## 变更编号

CHANGE-20260704-019

## 任务名称

新增生产 worker-watchdog 服务让 _recovery_watchdog_loop 在生产运行

## 需求出处

Phase 2 部署验证（main b4b5918 部署到生产）发现：PR #4 的 `mark_stale_worker_heartbeats` 已部署但 30 条 stale running worker_heartbeats 未被清理。根因：`backend/app/worker.py:1468-1470` 的 `_recovery_watchdog_loop` 启动条件为 `WORKER_TYPE in ("watchdog", "all")`，而 `docker-compose.prod.yml` 中 8 个生产 worker 容器的 `WORKER_TYPE` 均为具体类型（bars_scheduler/strategy_scheduler/calendar_scheduler/monitor_scheduler/strategy_batch/outbox/delivery/after_close_orchestrator），无任何容器匹配 `watchdog` 或 `all`，因此看门狗在生产从未启动。

## 修改前行为

- `_recovery_watchdog_loop` 每 60s 调用 `recover_stale_scheduler_job_runs` + `mark_stale_worker_heartbeats` 的代码逻辑正确（PR #4），但因启动条件不匹配任何生产 worker 容器，从未在生产运行；
- 生产 `worker_heartbeats` 表残留 30 条 stale running 僵尸记录（最早的来自 2026-07-02），导致 Worker 状态不可信；
- ALIGN-023 处于 KNOWN_GAP 状态，证据为"代码已实现，待生产部署验证"。

## 修改后行为

- 新增独立 `worker-watchdog` 生产服务（`WORKER_TYPE=watchdog`），命中 `worker.py:1469` 启动条件；
- 该容器运行 `_recovery_watchdog_loop`，每 60s 清理 stale scheduler_job_runs 和僵尸 worker_heartbeats（阈值 600s）；
- 预期部署后 60s 内 30 条 stale running 被标记为 stopped；
- ALIGN-023 仍保持 KNOWN_GAP（不关闭），待部署后验证 stale running 清零方可关闭。

## 影响模块

- 部署编排：`docker-compose.prod.yml` 新增 `worker-watchdog` 服务

- 不改 `backend/app/worker.py`（`WORKER_TYPE=watchdog` 已支持，`_recovery_watchdog_loop` 自带 heartbeat）

## 风险

- watchdog 容器失败会导致僵尸再次积累（`restart: unless-stopped` 兜底自动重启）
- 多容器并发安全：`mark_stale_worker_heartbeats` 使用 `UPDATE WHERE status='running' AND heartbeat_at < :cutoff`，幂等，即使未来多个 worker 兼任看门狗也不会重复标记
- `scripts/deploy.sh` 的 CORE_ONLY 模式不含 watchdog（与 monitor/outbox/delivery 一致），全量部署会启动
