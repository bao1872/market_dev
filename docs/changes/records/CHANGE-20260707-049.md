# CHANGE-20260707-049: Backfill Multiprocessing 优化

## 变更编号
CHANGE-20260707-049

## 任务名称
Feature Snapshot Backfill multiprocessing 并行优化

## 需求出处
PR #39 部署后 500-instrument 小样本验证耗时 28 分钟，估算全市场 5293 instruments × 121 trade_dates 约 6.6 小时。
用户要求评估提效方法并实施 multiprocessing 优化（用户选择 "A. 仅 multiprocessing (推荐)"，不做 compute-once-extract 重构）。

PR #41 初版方向正确但未通过 merge/deploy 门禁，用户在 TDD skill 下要求修 5 个 blocker（worker future 异常统计 / worker commit 失败统计 / per-date DB 异常 rollback / worker DB pool 过大 / workers 参数保护）。

## 修改前行为
1. `feature_snapshot_backfill.py` 只支持单进程模式（`backfill_instrument_first`），CPU 利用率低；
2. 单进程 commit 策略：整个 backfill 完成后一次性 commit（`main()` 控制），被 kill 中断会丢失所有未提交进度；
3. PR #41 初版 worker 循环中 `load_instrument_bars` 失败时只 rollback + 日志，不更新 `per_date_stats`，导致 `failure_rate` 计算不准确（load 失败的 instrument 不计入 failed）。

## 修改后行为

### v1（PR #41 初版）
1. 新增 `--workers N` 参数（默认 1 单进程，>1 启用 multiprocessing）；
2. 新增 `_worker_process_instruments()`：top-level 可 pickle 的 worker 函数，每个 worker 进程独立创建 `async_engine` + `async_sessionmaker`，per-instrument commit（被 kill 不丢已完成，resume 可续）；
3. 新增 `backfill_instrument_first_parallel()`：主进程创建/finalize run records，通过 `ProcessPoolExecutor` + `loop.run_in_executor` 分发 instrument chunks 到 worker 进程，`asyncio.as_completed` 收集结果；
4. worker 循环重构为三阶段：1) load_bars（失败 → 所有 trade_dates 标 failed + continue）；2) 逐 trade_date compute + upsert；3) per-instrument commit（失败 → rollback + 日志）；
5. `main()` 根据 `args.workers > 1` 分支：并行模式从 `get_settings().database_url` 获取 db_url 传给 worker，单进程模式保持不变。

### v2（[Blocker Fix] PR #41 第二轮）

**Blocker 1 — worker future 异常统计**：
- `asyncio.as_completed` 在 Python 3.12 返回 wrapper future，无法回溯原 chunk；
- 改用 `asyncio.gather(*futures, return_exceptions=True)`，按 chunk 顺序映射 `results[idx]` → `chunks[idx]`；
- worker 抛 `BaseException`（含 `KeyboardInterrupt`/`SystemExit`）时，对该 chunk 的每个 instrument × 每个 trade_date 增加 `failed`，避免 worker 崩溃但 run 仍 finalized 为 `succeeded`。

**Blocker 2 — worker commit 失败统计**：
- 旧：先 `success++` 再 `db.commit()`，commit 失败 rollback 后未修正 `success`；
- 新：改成 **per-date commit**——`upsert → db.commit() → success++`，commit 成功后才计 success；commit 失败 → `rollback + failed++`，DB 写入与 stats 严格一致。

**Blocker 3 — per-date DB 异常 rollback**：
- 旧：compute/upsert except 只 `failed++` 和 log，不 rollback，污染事务；
- 新：per-date except 中 `await db.rollback()`，每个 date 独立事务，后续 date 仍可继续成功。

**Blocker 4 — worker DB pool 过大**：
- 旧：`pool_size=5, max_overflow=10`，4 workers × 15 = 60 连接；
- 新：`pool_size=1, max_overflow=0, pool_pre_ping=True`，每 worker 只需 1 session，4 workers × 1 = 4 连接。

**Blocker 5 — workers 参数保护**：
- `--workers < 1` 直接 `parser.error()` 抛 `SystemExit`；
- `--workers > os.cpu_count()` 时 `warnings.warn()` + 自动 cap；
- 生产默认仍 1，建议先 `--workers 2` 小样本验证，再 `--workers 4`。

## 风险
1. **multiprocessing 隔离**：每个 worker 进程独立 `async_engine`，不能共享主进程连接池。`ProcessPoolExecutor` 需要 worker 函数可 pickle（`_worker_process_instruments` 为 top-level 函数，已验证）；
2. **per-date commit 性能**：相比 per-instrument commit 多了 N×D 次 commit 开销，但换来了：a) 被 kill 不丢已完成行；b) commit 失败不影响其他 date 事务（per-date 独立）。对 5000+ instruments × 121 trade_dates 的全量回补，per-date commit 额外开销可接受；
3. **生产验证未执行**：本 PR 只在 test DB 验证单元测试，生产 dry-run + 小批量验证需部署后执行；
4. **`asyncio.gather(return_exceptions=True)` 语义**：worker 抛 `BaseException` 时返回异常对象而非抛出，主进程通过 `isinstance(result, BaseException)` 检测，必须为每个 chunk 显式处理异常路径（已覆盖测试）。

## 遗留问题
1. **生产 multiprocessing 验证**：需部署后执行 `--workers 2 --dry-run` + 小批量 `--symbols` 验证，再扩大到 `--workers 4`；
2. **全量回补策略**：multiprocessing 部署后需评估实际提速倍数，再决定历史回补分批策略；
3. **compute-once-extract 优化**：本次不做（70% 因子非因果，需 truncate per trade_date），未来可探索对因果因子（ATR/BB/volatility）的预计算优化。
