## 1. 产品定位

盘迹（PanJi）V1.1 是面向 A 股个人投资者的多用户选股与盘中监控平台，核心能力：

- **每日选股**：基于趋势选股（DSA 趋势稳定性算法）的盘后全市场选股
- **盘中监控**：基于布林带 / Volume Node Cluster 的 1m bar 实时事件检测
- **消息投递**：通过飞书 Webhook / 平台应用将事件实时推送给用户
- **订阅体系**：邀请码注册 + 套餐制（`observe_20`=20 只监控 / `research_50`=50 只监控）+ 按自然月有效期，到期后允许登录但仅可续期（套餐契约见 `backend/app/constants/plan_contract.py`）

## 2. 用户角色（RBAC）

## 3. 用户流程

### 3.1 注册与登录

事实源：`backend/app/api/auth.py` + `backend/app/services/subscription_service.py` + `backend/app/constants/plan_contract.py`

```
注册：邀请码 → 校验 → 创建账户（active）+ 分配 member 角色 → 写入订阅记录（plan_code/monitor_limit 快照 + 按 grant_months 自然月计算到期日） → 生成 token
登录：email + password → bcrypt 校验 → 检查 status=active → 查询订阅状态 → 生成 access + refresh token
续期：登录态 + 邀请码 → 未到期从当前到期日顺延 grant_months 个自然月 / 已到期从当天计算 grant_months 个自然月 → 同时更新 plan_code/monitor_limit 为邀请码套餐快照
```

**套餐写入规则**（plan_contract）：
- 注册时从邀请码快照写入 `Subscription.plan_code` / `Subscription.monitor_limit`
- 续期时按邀请码套餐更新订阅 `plan_code` / `monitor_limit`（支持升级 observe_20 → research_50 或降级 research_50 → observe_20）
- 到期日按 `grant_months` 用 `dateutil.relativedelta(months=N)` 计算自然月（1月31日 + 1月 = 2月28日），兼容旧邀请码 `grant_days`（30 天近似）
- 旧邀请码/旧订阅由 Alembic 044 迁移回填 `plan_code='observe_20'`、`monitor_limit=20`、`grant_months=1`

**订阅到期处理**：
- 允许登录，返回 `subscription_active=false` + `next_route='/membership-expired'`
- 前端跳转 `/membership-expired` 续期页
- 仅可访问续期相关端点

### 3.3 自选股与监控

事实源：`backend/app/api/watchlist.py` + `backend/app/services/monitor_batch_service.py` + `backend/app/constants/plan_contract.py`

```
1. 用户在 /watchlist 加入自选股（active=true）
2. POST /watchlist 及恢复软删除记录前事务内校验 active count < monitor_limit（套餐额度）
3. 监控 Worker 在交易时段每 30 秒执行一轮监控
4. 监控范围 = 所有用户自选股去重 + watchlist_monitor 策略执行

```

**监控数量上限校验**（plan_contract）：
- 普通用户 `POST /watchlist` 及恢复软删除记录前必须在事务内校验 `active count < monitor_limit`
- 超限返回 `409 Conflict {"detail": "监控数量已达上限 N"}`（N 为当前套餐 monitor_limit）
- 管理员角色绕过监控数量限制（不校验额度）
- 无订阅记录的普通用户返回 `403 Forbidden`（无监控权限）
- 降级后已有数量超过额度**不删除**已有记录，只禁止新增（自然满足，仅校验新增）

事实源：`backend/app/api/public_beta.py` + `backend/app/services/beta_application_service.py` + `frontend/src/components/BetaApplicationModal.tsx`

```
1. 用户在落地页点击"申请内测"按钮（顶部导航 + Hero + Pricing 各套餐卡片）
2. 弹出 BetaApplicationModal 问卷弹窗（无需登录）：
   - 微信号（非必填，与手机号至少填一个）
   - 手机号（非必填，与微信号至少填一个）
   - 盯盘股票数量（必填正整数）
   - 使用理由（必选下拉：busy/too_many/forget/quant/other）
   - 其他补充说明（选"其他"时必填）
   - 隐私同意复选框（必勾）
3. 前端调用 POST /public/beta-applications 提交
4. 后端服务端校验 + IP 提取（SHA256 哈希存储） + 重复检测 + 频率限制
5. 写入 beta_applications 表 + 异步入队 Outbox 通知管理员飞书
6. 前端显示成功页面（申请编号 + 提交时间）
```

**校验规则**：
- 微信号/手机号至少填一个（服务端校验，DB 层均允许 NULL）
- 手机号格式：11 位数字（服务端正则校验）
- `reason_code` 必须为枚举值 `busy/too_many/forget/quant/other`
- `reason_code='other'` 时 `reason_other` 必填
- `watch_stock_count` 必须为正整数

**频率限制**：
- 同一 IP（ip_hash）1 小时内最多 5 次申请，超限返回 `429 Too Many Requests`
- 同一 phone/wechat 24 小时内重复提交返回原申请（`200 OK`，非 201）

**日志脱敏**：
- 日志输出手机号/微信号时只显示后 4 位（如 `***1234`）
- ip_hash 存储 SHA256 哈希，不存原始 IP
- API 响应仅返回 `{id, status, submitted_at}`，不返回完整联系方式

## 5. 交易日历判定规则（Mootdx 为唯一外部数据源）

事实源：`backend/app/services/mootdx_calendar_provider.py` + `backend/app/services/calendar_seed.py` + `backend/app/services/calendar_service.py` + `backend/app/models/calendar.py`

**核心变更（advice.md v9）**：
- 永久删除 Tushare 交易日历依赖，统一使用 `mootdx.utils.holiday`。
- 生产代码中只有 `mootdx_calendar_provider.py` 直接 `from mootdx.utils.holiday import holiday, holidays`。
- Pytdx 行情是否存在 K 线不再参与交易日判断。

**`trading_calendar` 状态语义**：

| 字段 | 说明 |
|------|------|
| `status` | `OPEN` / `CLOSED` / `UNKNOWN` |
| `source` | `MOOTDX_HOLIDAY` / `MOOTDX_HISTORICAL` / `MANUAL_OVERRIDE` |
| `is_trading_day` | 由 `status` 派生：`OPEN`→true，`CLOSED`→false，`UNKNOWN`→NULL |
| `verified_at` | Mootdx 校验时间戳 |
| `note` | 人工备注 |
| `validation_error` | 校验冲突/失败原因 |

**判定规则**：
1. **今天/未来日期**：调用 `holiday(YYYY-MM-DD)`，`False`→交易日（`OPEN`），`True`→非交易日（`CLOSED`），请求失败→`UNKNOWN`。
2. **历史日期**：使用 `holidays()` 返回的历史交易日集合核验；集合中存在→`OPEN`，不存在且 `holiday()` 为 `True`→`CLOSED`，冲突→`UNKNOWN` 并记录 `validation_error`。
3. **周六、周日**始终为 `CLOSED`。
4. `MANUAL_OVERRIDE` 不会被自动同步覆盖，除非带 `force=True`。

**降级链**（`is_trading_day_async`）：
1. DB 记录为 `OPEN`/`CLOSED` 时直接返回。
2. DB 为 `UNKNOWN` 或来源不可靠时查询 Mootdx。
3. Mootdx 失败时按周末日历兜底（仅判断是否为周末）。

**市场状态 API**（`GET /market/status`）新增诊断字段：
- `calendar_date` / `calendar_status` / `calendar_source` / `calendar_verified_at` / `degraded` / `degraded_reason`
- `UNKNOWN` 状态返回文案"交易日历待确认"，不得显示"休市"。

## 6. 业务边界

### 6.1 选股（Selector）

- **触发时机**：交易日 16:00 日线刷新完成后自动触发 DSA，18:30 兜底调度
- **执行位置**：后端 Worker（`strategy_batch`），不在 HTTP 请求内
- **状态机**：`queued → running → completed/partial_failed → published/failed`
- **自动发布门禁**：状态=completed + 覆盖率≥80% + failed_count=0
- **幂等键**：`strategy_key:strategy_version_id:trade_date:run_type:attempt_no`

### 6.2 监控（Monitor）

- **触发时机**：交易日 9:30-11:30 / 13:00-15:00，每 30 秒一轮
- **执行位置**：后端 Worker（`monitor_scheduler`）
- **监控范围**：所有用户自选股去重
- **事件冷却**：同一 `instrument_id + event_type + boundary` 在 600 秒内不重复写入
- **评估幂等**：`(策略版本, 股票, bar时间)` 唯一，`INSERT ON CONFLICT DO NOTHING`
- **session 聚合**：每个上午/下午只创建一条 SchedulerJobRun

### 5.3 消息投递（Delivery）

事实源：`backend/app/services/outbox_relay.py` + `backend/app/services/delivery_worker.py`

- **Outbox 模式**：业务写入与 Outbox 记录同事务，保证一致性
- **两阶段投递**：Outbox Relay 扩张为 MessageDelivery → Delivery Worker 实际投递
- **状态机**：`pending → sending → success/failed → retrying/dead`
- **重试退避**：`2^n * 30s`，最大 3 次
- **静默时段**：22:00-08:00 不投递，仅站内可见
- **消息组**：同一事件的 text+image 两条投递通过 `message_group_id` 关联
- **截图缓存**：按 `event_id + instrument_id + chart_version` 缓存 5-15 分钟

### 5.4 飞书渠道

事实源：`backend/app/services/feishu_webhook_adapter.py` + `feishu_platform_app_adapter.py` + `backend/alembic/versions/041_notification_channel_active_unique.py`

| 渠道类型 | 配置字段 | 适用场景 |
|---------|---------|---------|
| `feishu_webhook` | `webhook_url` + `sign_secret` | 群机器人（HMAC-SHA256 签名） |
| `feishu_platform_app` | `app_id` + `app_secret` + `receive_id` + `receive_id_type` | 个人用户（tenant_access_token） |

### 7.3 自选股涨跌幅规则

事实源：`backend/app/strategy_assets/manifests/watchlist_monitor.yaml`（outputs: `previous_close` / `change_pct`）+ `backend/app/strategy/monitors/watchlist_monitor.py`

自选股监控（`watchlist_monitor` v1.1.0）新增涨跌幅输出：

| 字段 | 类型 | 说明 |
|------|------|------|
| `previous_close` | number | 前一交易日收盘价（前复权） |
| `change_pct` | number (percent) | 当日涨跌幅（%） |

**计算口径**：
- `change_pct = (current_price - previous_close) / previous_close * 100`

- 涨跌幅着色按 7.1 规则（`change_pct > 0` 红、`< 0` 绿、`== 0` 灰）。
