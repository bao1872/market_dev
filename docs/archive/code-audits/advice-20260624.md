# 一、选股策略中，用户现在怎么设置筛选条件？

## 当前设计入口

用户目前需要点击**每列表头旁边很小的筛选图标**，弹出筛选框，再选择：

* 包含；
* 等于；
* 大于等于；
* 小于等于；
* 为空；
* 不为空。

多个列条件会被收集后发送给后端；页面也已经支持“全市场/我的自选”范围切换。

但现在这个功能**实际上没有完整打通**。

## 当前存在三个阻断问题

### 1. 前后端操作符不一致

前端支持：

```text
contains / eq / gte / lte / empty / not_empty
```

后端只接受：

```text
gt / gte / lt / lte / eq / between
```

所以用户打开数字列筛选框时，默认的 `contains` 会被后端以 422 拒绝；`empty` 和 `not_empty` 也同样不合法。

### 2. 有些被标记为“可筛选”的列，后端根本不允许筛选

后端只允许筛选策略 Manifest 中 `outputs.filterable=true` 的指标，例如：

* `dsa_dir_bars`
* `vwap_ret_avg`
* `vwap_ret_total`
* `offset_mean`
* `offset_std`
* `offset_variance_rate`
* `offset_percentile`
* `regime_value`
* `change_pct`

但前端把“股票”和“现价”等列也设置成了 `filterable=true`。其中：

* `stock` 不是指标；
* `price` 不在 DSA Manifest 输出白名单中。

用户筛选这些列会直接收到“非法 metric_key”。

### 3. 全文搜索在服务端分页模式下基本无效

表格保存了全文搜索文字，但向后端回调时只发送：

* page；
* pageSize；
* sort；
* filters。

没有发送全文搜索关键词。因此第一页里可能看似有搜索效果，但它无法对全量服务端结果进行正确搜索。

## 正确的产品设计

用户不能修改 DSA 算法参数，用户只能在**某个已发布批次的计算结果**上筛选：

```text
选择策略
→ 选择已发布批次
→ 选择股票池：全市场 / 我的自选
→ 设置结果筛选条件
→ 排序
→ 加入自选
```

筛选条件应当采用：

* 不同字段之间：**AND**
* 同一字段的多选值：**OR / IN**
* 不允许用户搭建跨策略 ALL/ANY 组合
* 不允许用户修改算法参数

建议把当前隐藏在表头的小筛选框，升级成表格上方明确的“筛选条件”区域：

```text
方向持续  ≥  [   ]
平均收益  ≥  [   ] %
偏移方差率 ≤ [   ] %
短期位置 介于 [ ] ～ [ ]
```

应用后展示为条件 Chip：

```text
方向持续 ≥ 50 ×
偏移方差率 ≤ 0.5% ×
短期位置 ≤ 30% ×
```

再增加轻量级“保存筛选视图”，例如：

```text
我的低位稳定股
我的高持续性结果
```

它只是保存：

```text
user_id
strategy_key
filters
sort
universe
name
```

不应重新引入 `SelectionPlan`。

### 必须修改

* 前端按字段类型生成合法操作符；
* `stock` 改成独立 `keyword` 参数；
* `universe` 保持独立参数；
* `price` 要么加入 Manifest 输出，要么取消筛选；
* Manifest 驱动列名、单位、操作符和默认显示；
* 删除 DSA Manifest 中仍残留的 `supports_plan_conditions: true`，因为 Plan 已经下线。

# 二、为什么自选股的上下节点、POC、最近触发都是空的？

## 后台监控任务实际上已经运行了

从截图看，13:00 时：

* 当前价已经出现；
* BB 上中下轨已经出现；
* 状态是“已计算”；
* 今日已有监控事件。

这证明主监控任务确实运行了。

代码中的实际触发方式是：

```text
交易日
09:30–11:30
13:00–15:00
每轮执行全部自选股并集
完成后等待 30 秒
```

每轮会取得最新已完成的 1 分钟 Bar，再结合日线和 15 分钟行情计算 BB、Volume Profile、节点和事件。

## 空值的主要原因不是“没有计算”，而是前后端字段类型不匹配

后台输出的节点不是数字，而是对象：

```json
{
  "price_mid": 67.12,
  "price_low": 66.85,
  "price_high": 67.39
}
```

`upper_node`、`lower_node`、`poc_price` 和 `last_touched_node` 都可能是这种对象。

但自选页面用了：

```typescript
toNum(metrics.upper_node)
toNum(metrics.lower_node)
toNum(metrics.poc_price)
toNum(metrics.last_touched_node)
```

JavaScript 对象无法转成有效数字，所以全部变成 `null`，最终显示 `-`。

首页则走了另一个错误方向，直接：

```typescript
String(object)
```

于是你看到了：

```text
[object Object]
```

## “最近触发”现在的定义也不对

后台的 `last_touched_node` 表示：

> 当前这一刻，价格是否落在某个节点价格区间里。

当前价不在任何节点区域时，它就应该是空的。它并不是“历史上最近一次触发的节点”。

真正的触发事件存储在 `StrategyEvent` 中。节点事件是在最后两根 1 分钟收盘价穿越节点价格时产生的，并带有：

* event type；

* boundary；

* current price；

* event time；

* instrument ID。

所以页面上的“最近触发”必须查询**最近一条 StrategyEvent**，不能读 `MonitorState.last_touched_node`。

## 正确修改方式

后端聚合接口应直接返回适合页面展示的投影：

```json
{
  "current_price": 67.42,
  "bb_upper": 79.12,
  "bb_mid": 71.20,
  "bb_lower": 63.28,

  "upper_node": {
    "price": 72.35,
    "low": 72.10,
    "high": 72.60
  },
  "lower_node": {
    "price": 65.80,
    "low": 65.55,
    "high": 66.05
  },
  "poc_price": 68.40,

  "latest_event": {
    "event_id": "...",
    "event_type": "node_cluster_touch",
    "event_time": "2026-06-24T13:00:00+08:00",
    "boundary": 68.26
  }
}
```

前端统一使用：

```typescript
node.price
```

表格里显示中间价，悬停时显示低点到高点。

此外，截图里的更新时间为 `05:00:59`，而页面时间是 13:01，这也是前端直接截取 UTC 字符串造成的。当前 `fmtTime()` 用正则直接截取 `HH:mm:ss`，没有转为上海时区。

应改成：

```typescript
new Date(value).toLocaleTimeString('zh-CN', {
  timeZone: 'Asia/Shanghai',
  hour12: false,
})
```

# 三、消息中心为什么看不到股票，也不能正确跳转？

这是当前最明显的**前后端消息契约错位**。

## 后端实际生成的类型

监控消息使用：

```text
MONITOR_MEMBER_EVENT
```

并且一条消息可能合并多只股票的事件。

## 前端仍然识别旧类型

前端只专门识别：

```text
monitoring_composite
selection_composite
process_event
system
```

这些还是被删除的“组合方案”时代的旧语义。

因此截图里才会直接显示原始英文：

```text
MONITOR_MEMBER_EVENT
```

策略列也是 `-`。

## 跳转逻辑也拿不到股票

前端只有在 `message_type === 'monitoring_composite'` 时，才去消息 body 顶层寻找：

```text
symbol
instrument_symbol
stock_symbol
```

但实际监控消息的股票信息被写进了 Markdown `items` 中，`resource_refs` 只有 event IDs 和 event types，没有结构化股票列表，所以无法得到 symbol。

而且一条合并消息可能包含三只甚至十只股票，直接跳到一只股票本身就不合理。

## 正确业务流程

一条监控消息应当是：

```text
监控事件集合
→ 打开消息详情
→ 展示本次触发的股票列表
→ 每只股票都可点击进入个股详情
```

推荐的结构化字段：

```json
{
  "message_type": "MONITOR_EVENT",
  "strategy_key": "watchlist_monitor",
  "strategy_name": "自选股监控",
  "resource_refs": {
    "instruments": [
      {
        "instrument_id": "...",
        "symbol": "300802",
        "name": "矩子科技",
        "event_ids": ["..."]
      }
    ]
  }
}
```

页面显示：

```text
自选股监控
矩子科技等 3 只股票触发 BB/节点事件
13:00
```

点击后打开消息详情抽屉：

```text
矩子科技 300802    节点穿越 68.26    查看个股
太辰光 300570      BB中轨穿越       查看个股
```

只有消息中恰好包含一只股票时，才可以直接跳转到个股详情。

## 仍然残留的两个问题

后台消息 Schema 仍然允许：

* `SELECTION_PLAN_SUMMARY`
* `MONITORING_PLAN_CONFIRMED`

说明旧 Plan 语义还没有清理干净。

另外，监控图片目前绕过 Outbox 直接发送，代码也留有 TODO。这会导致卡片和图片的重试、幂等和投递状态不一致。

还有一个更直接的运行错误：Worker 启动/异常通知构造 `NotificationMessageDTO` 时传了不存在的 `body` 字段，并使用不在允许枚举中的 `message_type="text"`。这个通知大概率始终发送失败，只是异常被捕获了。

# 六、当前业务主流程是否合理？

## 主方向是合理的

应保留以下结构：

```text
盘后行情完成
→ DSA 全市场统一计算
→ 发布不可变结果批次
→ 用户只做结果筛选
→ 加入自选

所有用户自选股求并集

→ 产生 StrategyEvent
→ 展开用户接收人
→ 创建 Message
→ Outbox
→ 站内/飞书投递
```

行情完成后自动检查覆盖率并触发 DSA 的代码已经建立，16:00 任务也按“全市场日线→覆盖率→DSA→15m→60m”执行。

## 不需要重写算法

`WatchlistMonitor` 虽然内部委托 BB Monitor 和 Volume Node Monitor，但这是算法模块化，不是用户可配置的多策略组合。这种实现可以保留。

需要重构的是**数据输出边界和页面职责**，不是算法。
