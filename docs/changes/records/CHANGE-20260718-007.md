## 2. 修改前理解

- **P0 阻断项（80 端口前端）**：
  - 生产入口固定 `80:80`，但 `frontend/Dockerfile` 误将构建产物 `COPY --from=builder /app/dist /usr/share`
    （非 nginx root），导致 `/usr/share/nginx/html` 下仍残留 nginx 默认欢迎页
  - 用户入口 `http://localhost/` 直接呈现 "Welcome to nginx" 而非盘迹 SPA
  - 前序会话错误地将此问题标记为 "out of scope"，继续合并流程
- **CI 诊断**：
  - 25 个 Ruff 新诊断（W292/I001/F401/C401/F541/F841）跨 10+ 文件
  - 4 个 mypy 新错误（attr-defined/arg-type/assignment）
  - 4 个 Architecture Rules 违规（duplicate-plan-feature-list）
  - 前序会话本地检查 "All checks passed" 但 CI baseline 对比捕获了新诊断

## 3. 修改目标

- **P0**：恢复 80 端口 SPA 服务，用户入口 80 必须直接呈现盘迹 SPA
- **CI**：所有 required CI checks 绿色，禁止带失败合并
- **长期**：新增静态测试守护部署合同，防止回归

## 4. 实际修改

### 4.1 P0：frontend/Dockerfile 修复（commit 1ea16d3）

```dockerfile
# 旧（错误）：
COPY --from=builder /app/dist /usr/share

# 新（正确）：
RUN rm -rf /usr/share/nginx/html/*
COPY --from=builder /app/dist/ /usr/share/nginx/html/
```

- `RUN rm -rf` 删除 nginx:alpine 自带的默认欢迎页（index.html / index.nginx-debian.html）
- `COPY` 目标改为 nginx root `/usr/share/nginx/html/`（与 nginx.conf `root` 指令一致）

根因分两类：**(A) 生产代码 schema_version 硬编码缺陷**；**(B) 测试未跟进生产重构**。

#### 4.4.1 生产缺陷修复（A 类，3 文件 4 处）

`feature_snapshot_service._SCHEMA_VERSION` 从 1→2→3 升级（CHANGE-004 Node Cluster engine）后，
消费侧仍硬编码 `schema_version == 1` 查询，导致新快照永远读不到——表现为
watchlist `metrics={}` / `calculation_status=WAITING_SNAPSHOT`、market_stocks `state_as_of=None`。

| 文件 | 位置 | 旧 | 新 |
|------|------|----|----|
| `backend/app/api/watchlist.py` | L362 snapshot 查询 | `== 1` | `== _SCHEMA_VERSION` |
| `backend/app/services/market_stocks_service.py` | L396 empty_state_as_of | `== 1` | `== _SCHEMA_VERSION` |
| `backend/app/services/market_stocks_service.py` | L487 snap_subq | `== 1` | `== _SCHEMA_VERSION` |
| `backend/app/services/market_stocks_service.py` | L554 state_as_of_dt | `== 1` | `== _SCHEMA_VERSION` |

两处均改为从 `feature_snapshot_service` import `_SCHEMA_VERSION` 常量，保证读写口径一致。

#### 4.4.2 stock_context `_empty_atomic_response` 修复（A 类，1 处）

`_empty_atomic_response` 在 `snapshot_missing` 场景（run 存在但 instrument 无 snapshot）
调用 `_build_data_quality(instrument, None, None, ...)` 传入 `run=None`，
导致 `hasSucceededRun=False`、`runTradeDate=None`、`runPublishedAt=None`——错误归零了
run 状态信息。修复：`_empty_atomic_response` 新增 `run` 关键字参数（默认 None），
`snapshot_missing` 分支传入 `run`；`no_published_full_run` 分支保持默认 None。
