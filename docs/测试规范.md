## 测试分类

项目仅保留以下三类测试。

### 1. Unit（单元测试）

- **定义**：针对单一函数、类、方法或算法的测试。
- **数据库要求**：**不得连接数据库**。
- **依赖要求**：使用普通 Python 对象、`Mock`、`AsyncMock`、`unittest.mock` 等替换外部依赖；不启动 FastAPI 完整依赖图（如不需要验证路由、依赖注入生命周期）。
- **适用场景**：
  - 纯计算函数（如日期转换、价格格式化、指标计算）
  - Service 层业务逻辑（当依赖 Repository 时使用 Mock）
  - 校验器、格式化器、枚举/常量定义
- **禁止事项**：
  - 创建 `create_async_engine`
  - `import aiosqlite`
  - 手写 `CREATE TABLE`
  - 使用 `db_session` fixture
  - 启动完整 FastAPI `TestClient` 来测纯函数

### 2. Integration（集成测试）

- **定义**：验证多个组件与真实基础设施（数据库、ORM、Service、API、事务、锁、约束）协同工作的测试。
- **数据库要求**：**必须使用 PostgreSQL 测试库 `bz_stock_test`**，测试结构必须来自 Alembic 迁移。
- **环境要求**：
  - `APP_ENV=test`
  - `TEST_DATABASE_URL` 必须指向 PostgreSQL，且库名以 `_test` 结尾
  - 测试 session 开始前自动执行 `alembic upgrade head`
- **适用场景**：
  - ORM 模型 CRUD 与关系行为
  - Service 层与数据库事务、锁、约束
  - API endpoint 与依赖注入（通过 `client` fixture）
  - 并发/竞态条件（如 invite code 并发兑换）
- **禁止事项**：
  - 使用 SQLite、aiosqlite、内存数据库
  - 手写 schema
  - 模块级 `db_session` fixture 覆盖公共 `db_session`
  - 在测试 seed 中复制套餐 features / monitor_limit 等业务值

### 3. E2E（端到端测试）

- **定义**：验证完整用户流程的测试，通常跨多个 API 调用和异步任务。
- **数据库要求**：使用**独立的 PostgreSQL 测试环境**（可与 Integration 测试库隔离，避免长时间运行污染）。
- **环境要求**：
  - 同样要求 `APP_ENV=test`
  - 测试前重置数据库并执行 `alembic upgrade head`
- **适用场景**：
  - 注册 → 登录 → 创建策略 → 运行 batch → 查看结果的完整流程
  - 涉及 Worker 或定时任务触发的异步流程
- **禁止事项**：
  - 使用 SQLite
  - 依赖手写 seed SQL 替代真实业务代码路径

## Fixture 使用规范

### 唯一事实源

`backend/tests/conftest.py` 是项目唯一数据库测试基础设施与 factory fixture 来源。所有 Integration 测试必须优先使用以下标准 fixtures：

| Fixture | 作用 |
|---|---|
| `pg_connection` | 原始 PostgreSQL 异步连接（底层调试或特殊场景） |
| `db_session` | 事务隔离的 AsyncSession，测试后自动回滚 |
| `app_db_override` | 覆盖 FastAPI `get_db` 依赖，使被测 API 使用测试 session |
| `client` | 基于 `app_db_override` 的 `AsyncClient` / `TestClient` |
| `user_factory` | 创建测试用户 |
| `role_factory` | 创建测试角色 |
| `subscription_factory` | 创建测试 subscription |
| `invite_code_factory` | 创建测试邀请码 |
| `instrument_factory` | 创建测试标的 |

> 注：`conftest.py` 已提供 `pg_connection`、`db_session`、`app_db_override`、`client`、`user_factory`、`role_factory`、`subscription_factory`、`invite_code_factory`、`instrument_factory` 等标准 fixtures；新增测试应优先使用。

### 禁止行为

1. **禁止模块 fixture 覆盖公共 `db_session`**
   - 任何测试模块不得定义名为 `db_session` 的 fixture。
   - 任何测试模块不得创建独立 engine 或 session factory 来替代 `db_session`。
2. **禁止在测试内手写 factory**
   - 创建用户、角色、标的等数据应使用 `conftest.py` 提供的 factory fixture，避免每个测试复制相似构造逻辑。
3. **禁止跨测试共享可变状态**
   - 不得把测试数据写入模块级变量并在多个测试间复用。

## 测试数据清理

### 基本原则

- **每个测试负责清理自己创建的数据**。
- 标准 `db_session` fixture 已提供事务隔离（nested transaction + rollback），测试结束后自动回滚，因此大多数测试无需显式删除。
- 对于需要显式 `commit` 的场景（如测试被测代码内部调用 `await db.commit()`），`conftest.py` 应使用 `join_transaction_mode=create_savepoint` 或等效 nested savepoint 方案，确保测试仍可回滚。

### 并发测试

- 并发测试必须使用**独立 session**，避免共享事务上下文。
- 并发测试必须**显式清理**自己创建的数据（或在独立 session 内使用独立事务，测试结束后回滚/删除）。
- 不得依赖 `db_session` 自动回滚来清理并发产生的外部副作用（如锁、通知、任务队列）。

## 环境要求

运行 Integration / E2E 测试前必须满足：

```bash
export APP_ENV=test
export TEST_DATABASE_URL=postgresql+asyncpg://user:pass@host:port/bz_stock_test
```

或 Alembic 使用的同步形式：

```bash
export DATABASE_URL=postgresql+psycopg://user:pass@host:port/bz_stock_test
```

### 强制检查

`backend/tests/conftest.py` 在导入时会校验：

1. `APP_ENV == "test"`，否则直接抛出 `RuntimeError`。
2. `TEST_DATABASE_URL` 非空。
3. 库名必须包含 `_test`，否则直接抛出 `RuntimeError`。

任何绕过上述校验自行创建 engine 的行为均属违规。

---

## 禁止 SQLite Integration 测试

- 所有被归类为 Integration 的测试必须使用 PostgreSQL。
- 项目中不存在“SQLite Integration 测试”这一分类。
- `test_config_validation.py` 中对 SQLite URL 的校验场景属于 Unit 测试，目的是验证 `config.py` 拒绝 SQLite，不属于 SQLite Integration 测试。

## 新增测试 checklist

在新增任何测试前，请确认：

- [ ] 已明确测试分类（Unit / Integration / E2E）。
- [ ] Unit 测试未连接数据库。
- [ ] Integration 测试使用 `db_session` / `client` 等公共 fixtures。
- [ ] 未在模块内定义 `db_session` fixture。
- [ ] 未使用 `sqlite`、`aiosqlite`、`CREATE TABLE`、`_SQLITE_DDL_STATEMENTS`。
- [ ] 未硬编码套餐数值（如 `monitor_limit=20/50`、`plan_code="observe_20"`），而是从 `plans` 表或服务查询。
- [ ] 未使用 `Role(name="user")` 或 `_ensure_role(db, "user")`。
- [ ] 未使用历史管理员套餐代码常量。
- [ ] 运行 `pytest tests/test_architecture_rules.py` 可通过。
