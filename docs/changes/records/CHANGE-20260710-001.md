# CHANGE-20260710-001: 飞书盘中高清实时截图（高清 + 不复用旧图/旧指标 + K线标题股票名称）

## 变更编号

CHANGE-20260710-001

## 任务名称

飞书盘中监控截图链路升级：高清截图、不复用上一轮旧图/旧指标、K线主标题显示股票名称。

## 需求出处

停止 PR #64（仅不合并、不部署，分支保持 OPEN）。从 `main` 重新开分支 `fix/feishu-realtime-hd-capture`，只做三件事：
1. 飞书截图变清晰（高清：viewport 1920×1200 + device_scale_factor=2，默认非 4）；
2. 盘中截图不复用旧图/旧指标（强 cache key + disable_cache + force_refresh + 实时 source_bar_time）；
3. K线图标题显示股票名称（如「宁德时代（300750）」）。

## 修改前行为

- 飞书截图使用 Playwright 默认视口（无显式 `device_scale_factor`，PNG 清晰度较低），不落库、不存 base64；
- 截图缓存 key 仅 `event_id + instrument_id + chart_version`，同一标的跨时点/跨周期可能命中旧图；
- `MonitorSnapshotService.get_snapshot` 与 `indicators` 接口无 `force_refresh` 旁路，飞书截图可能复用内存/Redis 中的旧指标缓存；
- Capture Snapshot 端点仅固定 1d 且 `include_realtime` 由常规开关控制，盘中 K线未必为当前实时；
- K线主标题显示股票代码，不显示名称。

## 修改后行为

- **高清截图**：capture worker 浏览器上下文使用 `viewport=1920x1200` + `device_scale_factor=2`（env `CAPTURE_VIEWPORT_WIDTH/HEIGHT` / `CAPTURE_DEVICE_SCALE_FACTOR`，默认 1920/1200/2，严禁 4 倍），提升 PNG 清晰度；截图为单张、不落库、不存 base64；
- **不复用旧图/旧指标**：
  - 截图缓存 key 维度扩展为 `event_id + instrument_id + chart_version + timeframe + source_bar_time + capture_run_id + device_scale_factor`，不同时间点/周期天然区分；
  - `disable_cache=True` 跳过读文件缓存但允许写最新缓存（飞书实时截图默认 True）；
  - `MonitorSnapshotService.get_snapshot(force_refresh=True)` 跳过内存缓存但写回最新；indicator 链路 `force_refresh=1&capture=1` 跳过 Redis 读缓存但写最新；
  - `source_bar_time` 优先取最新 `MonitorEvaluation.source_bar_time`（SUCCEEDED），兜底 `now_shanghai()` 分钟级；
  - Capture Snapshot 端点 `include_realtime=True` 且周期透传（1d/15m/1h/1w/1mo），`bars_limit` 按 `indicator_contract.INDICATOR_BARS` 对齐；
- **K线标题显示股票名称**：`StrategyChart` 主标题优先显示 `名称（代码）`，名称缺失回退代码；`CaptureStockPage` 按 URL `timeframe` 初始化（不锁日线）、请求 snapshot 携带 `force_refresh=1&source_bar_time=...`、状态栏展示 `data_source/is_partial/last_live_bar_time/quote status`。

## Follow-up 修复（2026-07-10 第二轮，阻断问题）

PR #65 评审发现的阻断问题，仅修阻断、不扩大范围、不 merge/部署。

### 阻断问题

1. `backend/app/api/capture.py` 虽新增 `timeframe`/`force_refresh`/`capture`/`source_bar_time`，但 `get_bars` 仍用 `timeframe=_CAPTURE_TIMEFRAME`（1d）且 `include_realtime=False`，导致截图仍是 1d 非实时；
2. `_df_to_responses` 仍传 `_CAPTURE_TIMEFRAME`，15m/1h 响应时间格式会错；
3. 前端 `CaptureStockPage` 读取 `snapshot.last_live_bar_time`，但后端实际返回 `bars.last_live_bar_time`；`endpoints.ts` 错误把 `last_live_bar_time` 放在 `CaptureSnapshotResponse` 顶层。

### 修复内容

- `backend/app/api/capture.py`：
  - `get_bars` 改为 `timeframe=timeframe` + `include_realtime=True`（硬规则，不再回退 1d/非实时）；
  - `_df_to_responses(df, instrument_id, timeframe)` 按 URL timeframe 格式化；
  - `compute_all_indicators(timeframe=timeframe, bars=bars_limit)`，其中 `bars_limit=INDICATOR_BARS.get(timeframe, CHART_BARS_COUNT)`；
  - 删除无意义 `realtime` 变量；新增实时聚合日志（timeframe/data_source/is_partial/last_live_bar_time/source_bar_time/force_refresh/capture）；
  - 新增测试 `test_capture_snapshot_realtime_timeframe_passthrough`：请求 `timeframe=15m` 时断言 `get_bars` 收到 `timeframe="15m"`+`include_realtime=True`、`compute_all_indicators` 收到 `timeframe="15m"`+`bars=INDICATOR_BARS["15m"]`、`_df_to_responses` 使用 `15m`、响应 `bars.timeframe=="15m"`。
- `frontend/src/api/endpoints.ts`：
  - 删除 `CaptureSnapshotResponse` 顶层 `last_live_bar_time`；
  - `BarListResponse` 补充 `last_persisted_bar_time` / `last_live_bar_time`（对齐后端 `BarListResponse` schema）。
- `frontend/src/pages/CaptureStockPage.tsx`：实时状态 `last_live_bar_time` 改从 `barsResponse`（`snapshot.bars`）读取；`is_partial`/`data_source` 本就来自 `barsResponse`。
- 文档：`docs/current/02-data-api-contracts.md` §4.1.1 明确 timeframe 透传硬规则 + `include_realtime=True` 硬规则；`docs/current/05-testing-acceptance.md` §3.8 增加 15m 透传阻断验收 + 前端验收。
