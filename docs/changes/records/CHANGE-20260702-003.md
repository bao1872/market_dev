## 2. 修改前理解

- 当前产品行为：`MarketDataAggregationService.get_bars()` 在入口和出口分别调用 `_cache_get()` 和 `_cache_set()` 进行 Redis 短缓存读写，但这两个函数未读取 `settings.bars_redis_cache_enabled` 开关。即使配置关闭缓存，测试和运行环境仍会读写 Redis，导致跨测试用例缓存污染。
- 当前前端流程：无变化。前端通过 `/instruments/{id}/bars` 和 indicator API 消费聚合结果，不直接感知缓存开关。
- 当前后端流程：`market_data_aggregation_service.py` 统一负责日线/周线/月线/15m/1h 聚合，Redis 缓存作为 5–15 秒短缓存层，但开关未生效。
- 当前数据/API/Worker/第三方关系：仅影响行情聚合服务内部缓存读写，不影响 DB、Pytdx、外部 Webhook 或 Worker 流程。
- 已阅读的当前设计文档：
  - `docs/README.md`
  - `docs/current/04-system-architecture.md`
  - `docs/current/06-backend-design.md`
  - `docs/current/08-api-contracts.md`
  - `docs/current/11-configuration-parameters.md`
  - `docs/current/18-code-doc-alignment.md`
- 发现的代码—文档—测试—生产冲突：`bars_redis_cache_enabled` 参数在 `11-configuration-parameters.md` 中已有定义，但 `market_data_aggregation_service.py` 的 `_cache_get`/`_cache_set` 未使用该参数，造成配置与实际行为不一致。

## 3. 修改目标

- 当前问题：`_cache_get`/`_cache_set` 忽略 `bars_redis_cache_enabled`，导致缓存开关关闭时仍读写 Redis，造成测试污染并掩盖缺失尾部补齐/降级逻辑。
- 修改后行为：当 `settings.bars_redis_cache_enabled` 为 `False` 时，`_cache_get` 直接返回 `None`，`_cache_set` 直接返回，不再访问 Redis。
- 验收标准：
  - `bars_redis_cache_enabled=False` 时，测试不再命中 Redis 缓存；
  - `bars_redis_cache_enabled=True` 时，原有缓存行为不变；
  - Phase 4 相关测试全部通过；
  - 新增 CHANGE 记录并更新 CHANGELOG。
- 明确不修改：
  - 不修改缓存 TTL、缓存键设计、序列化/反序列化逻辑；
  - 不修改行情聚合核心计算逻辑（DB 查询、Pytdx 补尾、复权、合成、partial bar 等）；
  - 不修改 DSA/Node Cluster/MACD/布林带公式参数；
  - 不修改除 `market_data_aggregation_service.py` 和文档外的其他文件。

## 5. 实际修改

- 代码文件：`backend/app/services/market_data_aggregation_service.py`
  - `_cache_get`：读取 `settings.bars_redis_cache_enabled`，若为 `False` 直接返回 `None`。
  - `_cache_set`：读取 `settings.bars_redis_cache_enabled`，若为 `False` 直接返回，不写入 Redis。
- 数据库迁移：无。
- 配置变化：无运行时配置变化。
- 兼容性和回滚：
  - 当开关为 `True` 时行为完全不变；
  - 当开关为 `False` 时，Redis 缓存不再读写，可安全回滚到提交前状态。
