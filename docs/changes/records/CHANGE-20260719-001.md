## 2. 修改前理解

### 2.1 复权因子路径分叉（§一）

`bars_scheduler_service._rebuild_factors_if_needed`（写入路径）和
`factor_consistency_audit.compute_expected_adj_factors`（审计期望值路径）虽然都计算复权因子，
但二者独立实现了 `_calculate_adj_factor`，且写入路径接受 `min_date` 参数（只计算受影响日期后的因子）、
审计路径计算全量序列——同一只股票的同一份 `bars_daily` + 公司行为输入下，两条路径会算出
**不同因子序列**。

**000688 失败案例**：rebuild 后 `before_hash == after_hash`（写入路径认为没有变化），
但 auditor 仍报告 288 mismatch（审计路径算出不同结果）。这是真实生产缺陷——
任何依赖 `factor_consistency_audit` 的告警都会因路径分叉持续误报。

CHANGE-20260718-007 §10.5 全市场审计的 614 不一致股票重建后 613 成功、1 失败（000688），
根因即此路径分叉。

### 2.2 12 算法族 broken entrypoints（§二）

CHANGE-20260718-007 S3.2 已诚实记录：12 个算法族中 7 个 `kernel_entrypoint` 指向不存在的
callable（`compute_smc_dto`/`compute_bollinger_bands`/`compute_sqzmom`/`compute_breakout`/
`compute_participation`/`compute_structural_features`/`compute_primary_secondary_relation`），
全部标为 `registered_only`。仅 macd 完成 wiring（`input_provider_wired`）作为参考实现。

S3.2 决策是"不重写 7 个 broken entrypoints，标 registered_only 诚实记录"。
§二 在此基础上推进：所有 12 算法族统一到 `canonical_adapters.py` 的 adapter 层，
`migration_status` 升级到新引入的 `production_wired`（更强断言：已是生产 canonical 入口，
禁止再退回 registered_only）。

### 2.3 四链实际未迁移到 compute_with_mdas

S3.2 明确声明：四条调用链（详情/盘后/盘中/Capture）当前仍直接调 MDAS + 各自 kernel，
绕过 `CanonicalComputationService`。`compute()` 零生产调用方。
§二 只完成 **adapter 层基础设施**——把签名各异的 kernel 包为统一 `(bars: pd.DataFrame, **kwargs)`
签名，并升级 migration_status。**四链实际迁移是后续工作**，本 PR 通过 AST 硬门禁
（`xfail(strict=True)`）锁定该目标。

#### 4.1.1 新增 `backend/app/services/adjustment_factor_calculator.py`（374 行）

提取唯一纯函数 `calculate_adjustment_factor_series`：

```python
def calculate_adjustment_factor_series(
    bars_daily_df: pd.DataFrame,
    corporate_actions: list[CorporateAction],
    *,
    algorithm_version: str = FACTOR_ALGORITHM_VERSION,  # "fq-v1"
) -> pd.DataFrame:
    """计算前复权因子序列（唯一真源，无 IO，无 min_date）。

    写入路径（rebuild_adj_factors）和审计路径（compute_expected_adj_factors）
    必须共用此函数，禁止任何路径独立实现 _calculate_adj_factor。

    Args:
        bars_daily_df: 必须含 trade_date/pre_close/open/high/low/close/volume 列
        corporate_actions: 公司行为列表（含 fenhong/peigu/peigujia/songzhuangu）
        algorithm_version: 算法版本（FACTOR_ALGORITHM_VERSION="fq-v1"）

    Returns:
        DataFrame: trade_date / adj_factor / source_event_id / fingerprint
                   fingerprint = SHA256(trade_date+adj_factor+source_event_id) 前 16 字符

    Raises:
        AdjustmentFactorDataError: 数据缺失（bars_daily_df 为空 / 缺列 / corporate_actions 缺字段）
    """
```

**设计原则**：
- **无 IO**：不查 DB、不读文件、不调 pytdx；输入纯 DataFrame + list
- **无 `min_date` 参数**：禁止调用方限定计算起点（消除路径分叉根因）
- **无 `supplement_df` 参数**：禁止调用方注入补充数据
- **无 `adapter` 参数**：禁止调用方替换算法实现
- **数据缺失抛 `AdjustmentFactorDataError`**：明确区分"算法失败"和"数据缺失"

#### 4.1.2 chanlunpro preclose 公式（与写入路径完全一致）

```python
preclose = (close_{D-1} × 10 - fenhong + peigu × peigujia) / (10 + peigu + songzhuangu)
```

除权因子 `f = preclose / open_D`，累积乘积得到 `adj_factor` 序列。

### 4.6 §五-D 修复 stock_context.py _SCHEMA_VERSION 硬编码（CHANGE-007 §4.4.1 遗漏）

#### 4.6.1 缺陷发现

§五-D 梳理 `_SCHEMA_VERSION` 明确常量时发现：`backend/app/api/stock_context.py:68` 本地定义
`_SCHEMA_VERSION = 1`，用于查询 `StockFeatureSnapshotRun.schema_version`（L93/L103/L122/L334）。
但生产写入 `StockFeatureSnapshotRun.schema_version = 3`（来自 `feature_snapshot_service._SCHEMA_VERSION = 3`）。

**影响**：`_find_latest_succeeded_run` 查询 `schema_version == 1` 的 run，但生产写入 `schema_version == 3`，
